# Context Snapshot Template · 语境快照模板

> [draft] 2026-07-01 v0.3
>
> 配套 [`80-meta/context-preservation.md`](../80-meta/context-preservation.md) 使用。
>
> **何时使用**：每次 commit 涉及 evolution.md 时（建议**所有**实质性 commit 都更新 evolution.md）。

---

## 模板（完整版）

```markdown
## v0.X · YYYY-MM-DD

### 触发 (Trigger)
什么具体事件 / 对话 / 错误让这个版本开始？
- [必填，1-2 句话]
- 例："用户提问：'如何增强研究信息的文件管理'"

### 目的 (Purpose)
为什么要做这个版本？不做会怎样？
- [必填，1-2 句话]
- 例："建立 3 层语境保存模型，让研究仓从档案柜变成活的系统"

### 当时的状态 (State)
- 已有：[文件、commit、工具]
- 缺：[验证、对比、实验]
- [必填，3-5 行]

### 方法 (Methodology)
为什么用这个方法？没选什么方法？为什么？
- [必填，2-3 句话]
- 例："选择 4 层一致性框架是因为它覆盖最广 + 已有体系支撑；没用纯软机制是因为软机制在 agent 群里会被腐蚀"

### 前提 (Assumptions)
基于什么预设？什么预设变了会让这个结论失效？
- [必填，2-3 句话]
- 例："假设 agent 群 > 3 个时治理才必要。如果未来是单 agent 主导，这个结论作废"

### 已知未知 (Known-Unknowns)
我还不知道什么？
- [必填，1-3 条]
- 例：
  - "我不知道多 agent 一致性协议具体怎么实现"
  - "我没验证 4 层框架是否覆盖所有一致性场景"

### 历史 (Lineage)
这个版本怎么从之前的版本演化而来？
- [必填，1-2 句话]
- 例："从 v0.2 治理框架的'决策反查'机制扩展而来；从 cron-as-system-mirror 的'已知未知'清单演化而来"

### 基调 (Tone)
这次研究是探索性 / 验证性 / 反对性？
- [必填，1 句话]
- 例："探索性。不预设 3 层模型一定有解，先写出来再说"

### 心境笔记 (Mind-State)
写 1-3 句关于"我当时的心境"。
- [可选，1-3 句话]
- 例："今天用户问了一个我之前没想清楚的问题。写完这份文档后我自己对'语境'的理解也更清楚了"

### 做了什么 (What was done)
具体做了什么（文件清单 + commit 链接）
- [必填，bullet 列表]

### 未来伏笔 (Future Threads)
预判下一步可能的研究方向（不展开，只是占位）
- [可选，2-5 条]
- 例：
  - "自动生成 Layer 1 心境笔记的机制"
  - "跨 session 语境共享协议"
  - "语境保存 vs 隐私的张力"

### 反方观点 (Counter-View)
如果有人反对这个版本，最可能反对什么？
- [可选，1-2 句话]
- 例："可能反对：语境保存会变成日记噪音。需要验证 8 维度的最小有效集"
```

## 简化版（轻量使用）

如果觉得完整版太重，可以用简化版：

```markdown
## v0.X · YYYY-MM-DD

**触发**：<一句话>
**目的**：<一句话>
**做了什么**：<bullet 列表>
**未解决**：<1-3 条>
**心境**：<1 句话，可选>
```

## 字段必填 / 可选规则

| 字段 | 必填 / 可选 | 适用场景 |
|---|---|---|
| 触发 | 必填 | 所有 |
| 目的 | 必填 | 所有 |
| 当时的状态 | 必填 | 所有 |
| 方法 | 必填 | 实质性变更 |
| 前提 | 必填 | 推翻旧结论时 |
| 已知未知 | 必填 | 所有 |
| 历史 | 必填 | 所有 |
| 基调 | 必填 | 所有 |
| 心境笔记 | 可选 | 个人项目时建议填 |
| 做了什么 | 必填 | 所有 |
| 未来伏笔 | 可选 | 重要版本时建议填 |
| 反方观点 | 可选 | 颠覆性变更时建议填 |

## 怎么用

### 步骤 1：commit 前写快照
在 commit 之前，把当前版本的语境快照**先写下来**（不是 commit 之后再补）。

### 步骤 2：把快照放到 evolution.md
按时间顺序追加到 evolution.md。

### 步骤 3：commit 时引用快照
commit message 里可以引用 evolution.md 的版本号，方便回溯。

### 步骤 4：定期 review 语境
- 每月看一次 evolution.md 的所有版本
- 哪些版本"语境丰富"（写满了 8 维度）？哪些"语境贫乏"（只写了做了什么）？
- 贫乏的版本往往是"惯性推进"的——下次要补上

## 8 个维度的反模式

| 反模式 | 表现 | 修正 |
|---|---|---|
| 把"做了什么"当语境 | 只写 commit 列表 | 至少加"目的"和"已知未知" |
| 写得太长 | 一次写了 1000 字心境 | 心境最多 3 句，扩展放单独文件 |
| 把"前提"当废话 | 留空或写"无前提" | 至少写"基于 X 假设" |
| "已知未知" 写成"已知" | 写"我们知道 X" | 必须是真的"不知道" |
| "未来伏笔" 写成"未来计划" | 写具体的 TODO | 伏笔是占位，不是承诺 |
| 不写"基调" | 留空 | 必须填，因为影响"怎么读这个文档" |
