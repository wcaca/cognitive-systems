# 递进认知链 · 边跑边暴露作为研究方法

> 2026-07-01 · v0.8.1 元观察
> 起源: sas-graph v0.6.2-0.6.7 跨项目推进 (6 顿悟 A-F)
> 核心命题: **6 个顿悟不是独立的, 是 1 个递进认知链**
> 框架影响: 反哺 v0.6 (insight-to-work) + v0.8 (IPL) 的改进

## 现象

2026-07-01 晚, 我在 sas-graph 跑完 6 个子版本 (v0.6.2-0.6.7), 暴露了 6 个"顿悟":

| 顿悟 | 暴露位置 | 暴露的问题 |
|---|---|---|
| A | 写 getVerbDistribution | verb 在 Graph 里出现 2 次 (节点 + 边), 算错 |
| B | 写 JSON 序列化 | Date.now() 让 seed non-idempotent |
| C | 写完 3 个 commit | 用户拿到仓库不知从哪开始 |
| D | 测 createdAt 冲突 | merge 后时间线信息丢失 |
| E | 写 Dijkstra | 有向图上反向边也算可走 |
| F | 收尾 tsc --noEmit | vitest 通过但 tsc 报错 |

**乍看是 6 个独立问题**. 但**仔细看暴露顺序**, 它们**不是独立的**:

```
A schema 模糊 (写代码时暴露)
  ↓ 让 B 关注: "时间数据是 schema 一部分吗?"
B 时间漂移 (跨调用暴露)
  ↓ 让 C 关注: "数据可复现, 但用户用不上 = 没意义"
C 部署不可见 (跨用户暴露)
  ↓ 让 D 关注: "跨用户 = 跨进程, 跨进程 = merge, merge = 3 层"
D 跨进程 3 层 (跨进程暴露)
  ↓ 让 E 关注: "跨进程算法 = 算法一致性问题"
E 算法一致性 (算法层暴露)
  ↓ 让 F 关注: "runtime 一致, compile-time 呢?"
F compile-time 盲点 (工具链层暴露)
```

**6 步, 6 个层, 1 个链**.

## 6 顿悟的"层"分布

每个顿悟暴露的是**不同抽象层**的真问题:

| 顿悟 | 抽象层 | 问题类型 | 检测维度 |
|---|---|---|---|
| A | **数据 schema** | 字段重复 | runtime (聚合算错) |
| B | **时间语义** | 演示数据 vs 事件数据混淆 | runtime (跨进程不可复现) |
| C | **用户接口** | 部署后用户不可见 | 部署 (用户行为) |
| D | **跨进程语义** | schema/语义/审计 3 层缺位 | runtime (跨进程) |
| E | **算法一致性** | 抽象边界 (有向 vs 无向) | 设计选择 |
| F | **工具链** | compile-time 盲点 | compile-time |

**规律**: 暴露顺序 = **抽象层从低到高**:
- 1-3: 单进程内 (schema / 时间 / 部署)
- 4-5: 跨进程 (merge / 算法)
- 6: 跨工具链 (compile-time)

**这是"边跑边暴露"的内在结构**——**不是随机暴露, 是抽象层爬升**.

## "边跑边暴露"作为研究方法

传统研究方法 (top-down):
- 假设 → 设计 → 验证
- 一次设计完整, 然后跑

边跑边暴露 (bottom-up):
- 跑 → 暴露 → 假设 → 再跑
- **每个暴露的假设, 重塑下一个搜索方向**

**优势**:
- 暴露的是**真实**问题 (不是设计时的"假想"问题)
- 暴露的顺序揭示了**抽象层爬升路径** (教科书不会告诉你这路径)
- 每个暴露的顿悟都**直接对应代码可改进点** (不是抽象"建议")

**代价**:
- 不能跳步 (跳步 = 漏层)
- 速度慢 (5 步 = 25 分钟)
- 容易"伪暴露" (看似暴露, 实际是设计选择, 如 E)

## 对 v0.6 / v0.8 框架的反哺

### 1. v0.6 (insight-to-work) 反哺

v0.6 的 4 步桥接 (顿悟→验证→4 面→制品) 在仓内 prototype 跑通 (v0.6.5 / v0.7).
但**没显式包含"暴露"环节**——顿悟是输入, 不是产出.

**改进**: 加第 5 步 "**暴露**" (4 → 5 步):
- 顿悟 → 验证 → 4 面 → 制品 → **暴露**

"暴露"环节的内容:
- 在制品完成后, 主动问"还有什么没暴露?"
- 测边界 / 测反向 / 测集成 / 测 compile-time
- 把"暴露"当**制品的一部分** (顿悟文档 = 制品的副作用)

### 2. v0.8 (IPL) 反哺

v0.8 的 9 阶段框架在 v0.6.2-0.6.6 完整跑通.
但**阶段 5 (实施) 的 PIVOT echo 缺细化**——"5 步一回"是经验值, 不是原理.

**改进**: 阶段 5 加"**抽象层爬升**"维度:
- PIVOT echo 时问 "我现在在哪个抽象层?"
- 抽象层从低到高: schema → 时间 → 部署 → 跨进程 → 算法 → 工具链
- **每跨 1 个抽象层 = 1 个 PIVOT echo 节点** (不是固定 5 步)

**实证**:
- v0.6.1 推进: 1 个 echo (跳 2/3/9) = 0 顿悟
- v0.6.2-0.6.6 推进: 6 个 echo (每个子版本 = 1 抽象层) = 6 顿悟
- **echo 节点数 = 抽象层数 = 顿悟数** (这是规律)

### 3. 27 点 rubric 反哺

cognitive-systems/80-meta/insight-completion-rubric.md 当前 3 层 × 3 子分 = 27 点.
**漏了 1 个维度: compile-time 验证**.

**改进**: L3 加第 4 个子分 = "**type-checked**" (8 维度 → 9 维度 = 32 点):
- 之前: 测 runtime 通过, 自评 8 分
- 之后: runtime + compile-time 都过, 才 8 分

**实证**:
- v0.6.5 merge 测覆盖了 runtime, 没覆盖 tsc
- v0.6.7 收尾暴露 F = vitest 61/61 过但 tsc 报错
- **加 type-checked 维度后, 这种盲点会被提前发现**

### 4. 阶段 8 (监控) 反哺

v0.8 IPL 阶段 8 (监控) 在 v0.6.2-0.6.6 显式 N/A.
但**顿悟 F 揭示了一个新的监控层: compile-time**.

**改进**: 阶段 8 加 "**工具链监控**" 子层:
- runtime 监控 (传统)
- LLM 监控 (cron-as-mirror 已有)
- **compile-time 监控** (新) = CI 必须跑 tsc / eslint / type check

## 元观察的元观察

我做 v0.6.2-0.6.6 时, **没显式遵循这 6 层爬升规律**——是事后才抽象出来.
但**事后抽象出来后, 它能预测 v0.6.8+ 的暴露方向**:

| 抽象层 | v0.6.2-0.6.7 已暴露 | v0.6.8+ 待暴露 |
|---|---|---|
| schema | A (verb 双源) | type union 边界 (更细) |
| 时间 | B (idempotent) | timezone / 时区 |
| 部署 | C (代码 ≠ 可跑) | npm publish / 版本兼容 |
| 跨进程 | D (merge 3 层) | conflict resolution 算法 |
| 算法 | E (有向 vs 无向) | graph isomorphism (子图匹配) |
| 工具链 | F (vitest ≠ tsc) | bundler / runtime 差异 |

**这 6 个 v0.6.8+ 暴露方向, 是从"6 顿悟 = 6 抽象层"规律推出来的**.

## 教训

1. **"边跑边暴露" 揭示的是抽象层爬升路径, 不是问题列表** — 这是研究方法的核心
2. **顿悟之间有依赖** — A 让 B 关注时间, B 让 C 关注用户, C 让 D 关注跨进程
3. **每跨 1 个抽象层 = 1 个新维度** — 不是 6 步 6 步, 是 1 链 1 链
4. **框架应该能预测下 1 个暴露** — 规律抽象后, 框架不是被动, 是主动
5. **"暴露" 应该成为制品的一部分** — 顿悟文档 = 制品的副作用, 不是额外产出
