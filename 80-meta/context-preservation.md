# Context Preservation · 研究语境的保存

> [draft → active] 2026-07-01 v0.3
>
> **核心命题**：研究的"语境"不是装饰，是研究本身的一部分。
> 没有语境保存的研究是"无根之木"——后来人（包括未来的自己）只能看到结果，**看不到当时怎么想、为什么这么想、还卡在哪**。
>
> **本文目标**：建立一套"研究语境"的保存模型，让研究仓从"档案柜"变成"活的系统"。

---

## 一、为什么需要语境保存

### 现状（v0.2 之前）

cognitive-systems 仓（截至 v0.2.1）的 evolution.md 记录了：
- 做了什么
- 起源对话主题
- 状态（文件数、字数）
- 仍欠缺的

**但没记**：
- 当时为什么做（不是"起源对话主题"，是"内心深处的动机"）
- 当时心智状态（累 / 清醒 / 兴奋 / 低落）
- 当时踩了哪些坑、绕了哪些路
- 当时觉得"什么最重要"（事后看可能不是）
- 当时觉得"什么最不确定"

### 结果

未来回看 v0.1 / v0.2 时，只能看到"做了什么"，看不到"为什么这么做、当时怎么想"。**这是研究价值的最大浪费**——因为研究的真正价值往往在过程里，不在结果里。

### 一个具体例子

`consistency-governance.md` 写了"4 层一致性 + 硬约束/软机制/反向防御"。
但没写"为什么把硬约束放第一位、为什么不用纯软机制"。
未来回看时，会觉得这个分层理所当然——**但它不是**。它是在我们聊完"agent 行为一致性治理"那个具体对话后才浮现的。

如果当时写下了"为什么"，未来就**能复现当时的推理链**，而不是只看到结论。

## 二、语境的 8 个维度

每个研究都有语境，至少包含 8 个维度。**一个都不能省**——少一个维度，研究就缺一根筋：

| 维度 | 内容 | 没保存会怎样 |
|---|---|---|
| **目的** | 为什么要研究这个 | 后来人不知道为什么 |
| **触发** | 什么让我现在开始研究 | 不知道为什么是"现在" |
| **状态** | 已有什么、还缺什么 | 后来人重复做 |
| **方法** | 为什么用这个方法 | 不知道可不可以换 |
| **前提** | 基于什么预设 | 前提变了结论作废 |
| **未知** | 已知未知是什么 | 假装问题不存在 |
| **历史** | 怎么走到这里 | 不知道演进路径 |
| **基调** | 探索性 / 验证性 / 反对性 | 不知道该怎么读 |

### 维度详解

**目的 (purpose)**
- 不是"研究什么"（这是主题）
- 是"为什么要研究这个、不研究它会怎样"
- 比如："研究 agent 一致性治理，因为 v0.1 之后我意识到 agent 群会腐蚀系统"

**触发 (trigger)**
- 触发 = 什么具体事件 / 对话 / 错误让研究开始
- 比如："用户问了一个具体问题" / "v0.1 暴露了一个 bug" / "看到一个 tweet 启发"

**状态 (state)**
- 已有：哪些文件、哪些 commit、哪些工具
- 缺：哪些验证、哪些对比、哪些实验
- 比如："v0.1 有 6 个文档，缺审计跑过的实例"

**方法 (methodology)**
- 为什么用这个方法
- 比如："选择 4 层一致性框架是因为它覆盖最广 + 已有体系支撑"

**前提 (assumptions)**
- 基于什么预设
- 比如："假设 agent 群 > 3 个时治理才必要"

**未知 (known-unknowns)**
- 明确说"我还不知道什么"
- 比如："我不知道多 agent 一致性协议具体怎么实现"

**历史 (lineage)**
- 这个研究怎么从之前的研究演化而来
- 比如："从 cron-as-system-mirror 的'已知未知'清单演化而来"

**基调 (tone)**
- 探索性：开放、不预设结论
- 验证性：在测一个具体假设
- 反对性：在反驳某个已有结论
- 比如："探索性，不预设 cron 设计一定有解"

## 三、三层语境保存模型

提出一个**三层语境保存**：

```
┌──────────────────────────────────────┐
│  Layer 3: 演化语境（versioned）       │
│  每次 commit 时的语境快照             │
│  → evolution.md + context-snapshot   │
├──────────────────────────────────────┤
│  Layer 2: 会话语境（linked）          │
│  触发该研究的具体对话                 │
│  → 链接到原始对话的引用               │
├──────────────────────────────────────┤
│  Layer 1: 一次性语境（ephemeral）     │
│  研究当下的心境、注意力、灵感         │
│  → 自动生成、自动归档                 │
└──────────────────────────────────────┘
```

### Layer 1：一次性语境（ephemeral）
当下研究的"现场"：
- "我今天累了，这个 v0.2.1 是趁着清醒写的"
- "用户刚做完一个 3 周项目，情绪在恢复期"
- "LLM 最近一次升级在 prompt 遵循性上有变化"

**特征**：不写下来就丢失，但写多了又成噪音。
**解法**：结构化的快照，不是流水账。

落地：每次 commit 时的"心境笔记"，3-5 句话，必填。

### Layer 2：会话语境（linked）
研究不是凭空开始的——它有触发对话。**链接**到原始对话（不是复制）。

落地：evolution.md 每个版本加一行：
```
触发对话：[conversation-id] @ 2026-07-01
关键引文："> 这里是被用户问的具体问题"
```

链接是单向引用（指向 conversation），不是双向同步（对话本身不需要引用 evolution）。

### Layer 3：演化语境（versioned）
每个 commit 时的"全状态快照"。这其实是**增强版 evolution.md**——除了"做了什么"，还记"为什么、当时怎么想、未解决什么"。

落地：每个 v0.X 段都按 8 维度结构写（详见 [context-snapshot-template](../90-conventions/context-snapshot-template.md)）。

## 四、关键张力（4 个）

这些是必须面对的取舍，**没有银弹**：

### 张力 1：语境保存 vs 噪音
**问题**：保存太多语境 = 仓变日记；保存太少 = 失去意义。
**平衡点**：
- 每次 commit 必填的"最小语境"（3-5 句话）
- 可选填的"扩展语境"（长篇心境笔记）
- L0 read / L1 append 的权限边界：语境只是补充，不能替代理性思考

### 张力 2：预判 vs 适应
**问题**：过度预判 = 提前把未来研究做死，失去灵活性；不预判 = 每次研究都从零开始。
**平衡点**：
- 预判只做"接口预留"，不做"内容填充"
- "未来研究方向（占位）"段可写，但展开是后续 commit 的事
- 预判错了允许"撤回"（不写成主文档，而是写在 meta 段）

### 张力 3：链接 vs 复制
**问题**：引用对话 vs 复制对话内容。
**平衡点**：
- 链接 + 5-10 句的"原话引用"
- 全文链接，但仓内不复制大段对话
- 这样对话隐私保留 + 关键语境不丢

### 张力 4：过程记录 vs 成果展示
**问题**：仓是给未来"自己"看的，还是给"公众"看的？
**平衡点**：
- 分两层：
  - `80-meta/` 下放过程记录（自己看）— 包含心境、探索、失败
  - 主目录放成果展示（公众看）— 包含结论、协议、模式
- `80-meta/` 是"研究日记"，主目录是"研究成果"

## 五、跟已有体系的连接

这 4 层语境不是孤立的，跟之前的研究都连着：

| 语境保存 | 一致性治理 | cron 镜子 | 仓结构 00-90 |
|---|---|---|---|
| 语境保存 | 决策反查的"决策时刻"是什么 | cron 任务的"启动语境"是什么 | 00-essence 是仓的"宪法语境" |
| 趋势预判 | 协议演进的下一步会是什么 | cron 任务的下一次失败会是什么 | 90-conventions 怎么自适应 |
| 文件管理 | 治理文件怎么"自维护" | cron 怎么"自调度" | 00-90 怎么不被侵蚀 |

**这是一张网**——语境保存是这张网的"主线"。

### 具体连接到 consistency-governance

`consistency-governance.md` 里"决策反查"机制正是**语境保存的具体应用**：
- 反查要求写"旧判断是什么、新证据是什么、新判断是什么"
- 这就是 Layer 3（演化语境）的子集
- 把"决策反查"扩展到"语境反查" = 把治理框架升级

### 具体连接到 cron-as-system-mirror

`cron-as-system-mirror.md` 第 4 层"harness 的已知未知"清单正是**语境保存**：
- 已知未知 = 当下不知道什么
- 保存这个清单 = 保存 Layer 1 的一次性语境
- 升级 cron prompt 让它每次启动时记录"今天我又发现了什么新的未知"

### 连接到 00-90 结构

- 00-essence 是"仓的宪法语境"（最稳定的语境）
- 90-conventions 是"治理语境"（约束）
- 80-meta 是"过程语境"（本次研究的现场）
- 这三层正好对应 L3 / L2 / L1 语境保存模型

## 七、开放问题

以下问题还没有满意答案：

1. **怎么自动生成 Layer 1 一次性语境？**
   - 现在靠人写
   - 目标：harness 自动从对话历史抽取关键心境
   - 难点：心境是主观的

2. **怎么验证语境保存的"完整性"？**
   - 现在靠人 review
   - 目标：有 checklist 能验证"这 8 个维度都填了"
   - 难点：怎么避免变成形式主义

3. **语境跟记忆的关系是什么？**
   - 语境是一次的，记忆是长期的
   - 哪些语境应该升级为记忆？
   - 哪些记忆应该降级为语境？

4. **跨 session 怎么共享语境？**
   - 现在每个 session 独立写 evolution
   - 目标：cron session 跟 root session 共享"今天研究了什么"
   - 难点：跨 sandbox 的语境共享（已知 creation-loop v0.3 限制）

## 八、一句话总结

> **研究的语境 = 研究的过程 + 状态 + 心境 + 未知 + 历史**。
> 
> 没有语境的研究是"无根之木"。有语境的研究是"活的系统"——未来回看时，能**复现当时的推理链**，而不仅看到结论。
> 
> 把语境保存变成机制 = 把研究仓从"档案柜"升级为"活的认知系统"。
