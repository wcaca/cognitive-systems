# 顿悟驱动的项目生命周期 (IPL · Insight-driven Project Lifecycle)

> 2026-07-01 · v0.8 框架 (seed, 待更多项目实证)
> 起源: sas-graph v0.6 → v0.6.1 跨项目推进测试 (`wcaca/sas-graph`)
> 上承: `insight-to-work.md` (4 步桥接 = IPL 的阶段 4-7 浓缩)
> 核心命题: **顿悟不是工作流, 是质量门**

## 为什么需要 v0.8

v0.6 的 4 步桥接 (顿悟→验证→4 面→制品) 在 **cognitive-systems 仓内 prototype** 跑通 (v0.6.5 / v0.7)。
但推到**真实 GitHub 项目** (sas-graph) 时暴露缺口:

- 我直接进 4 面展开, **跳过了范围/规划** (阶段 2+3)
- 实施时**没 PIVOT echo** (中途边数写错 2 次)
- 完成后**回顾在 commit msg 一笔带过** (没正式 retrospective)

→ **半跑 vs 全跑的质量差异**: 同样 8.8 分, 完整 9 阶段能到发布, 半跑只能 prototype.

v0.8 = 把"顿悟嵌入每个阶段出口"作为质量门.

## 9 阶段 + 9 顿悟检查点

### 阶段 1: 愿景 (WHY) · 1 个问题
**顿悟检查**:
- 一句话能说清这个项目存在的原因吗?
- 不做世界会怎样? 做了世界会怎样?
- 跟"看起来很酷" vs "真问题" 的差距在哪?

**反模式**: "做个东西看看" / 没有 WHY / 把手段当目的

### 阶段 2: 范围 (WHAT) · 2 个清单
**顿悟检查**:
- **Must (≤5)**: 不做这 N 项, 项目就不成立
- **Not (≤3)**: 明确排除 (防范围蔓延)
- **Maybe (≤5)**: 留口子但不当承诺

**反模式**: 范围蔓延 / "做完整的 N 版本" / 怕做错就什么都做

### 阶段 3: 规划 (WHEN) · 3 杠杆排序
**顿悟检查** (按 3 杠杆选下一步):
- **阻塞下游最高**: 谁等这个? 等多久?
- **长期价值最高**: 3 个月后还重要吗?
- **立刻可做**: 今晚能开干吗? (避免过度准备)

**反模式**: 挑最炫的做 / 顺序错位 / 永远在准备

### 阶段 4: 设计 (HOW) · 4 面展开
**顿悟检查**:
- **历史**: 这东西怎么来的? (起源/演化/前驱)
- **结构**: 在更大体系里什么位置? (上下位)
- **应用**: 谁用、怎么用、用在哪场景?
- **限制**: 不适用场景、边界、性能瓶颈

**反模式**: 想到一个点就开干 / 4 面不全 / 把限制当 bug

### 阶段 5: 实施 (DO) · PIVOT echo
**顿悟检查**:
- **每 5 步 echo 1 次 WHY** (5 步 = 1 个 function / 1 个 commit / 1 个段落)
- 中途发现 PIVOT 错了, **立刻停** (不是"写完再回头看")
- 代码写一半走神 = PIVOT 掉线征兆

**反模式**: 埋头 1 小时不抬头 / 写完才发现自己跑偏 / "差不多就行"

### 阶段 6: 验证 (CHECK) · 27 点 rubric
**顿悟检查** (3 层 × 3 子分):
- **L1 (insight core)**: 9 分够清晰吗?
- **L2 (4 faces)**: 4 面都覆盖了吗?
- **L3 (artifacts)**: 完成度 + 集成度 + 演化痕迹
- **阈值**: prototype 6.5 / 科研 8.5 / 发布 9.0

**反模式**: "跑通就完事" / 主观"看起来对" / 没阈值

### 阶段 7: 部署 (SHIP) · 风险点清单
**顿悟检查**:
- **SSH 风险** (server 端 sync 失效已知)
- **Secret 暴露** (.env 误提交)
- **路径差异** (绝对 vs 相对)
- **环境差异** (dev vs prod 端口/版本)
- **rollback 计划** (出问题能 5 分钟内回滚吗?)

**反模式**: 改完直接 push / "线上看" / 不写 rollback

### 阶段 8: 监控 (WATCH) · cron-as-mirror 6 层
**顿悟检查** (mirror 的 6 层依赖是否都健康):
- **基础设施层**: 磁盘/端口/服务进程
- **LLM 层**: API quota / 模型升级 / 限速
- **prompt 层**: template 漂移 / 变量缺省
- **元认知层**: 框架本身是否还适用
- **人机耦合层**: 用户行为变化 / 期望漂移
- **反馈闭环层**: alert 有效性 / 噪声比

**反模式**: "上线就不管" / alert 太多 = 没 alert / 监控指标≠真实健康

### 阶段 9: 回顾 (REVIEW) · 8 维度 retrospective
**顿悟检查** (按 8 维度格式):
- **触发**: 是什么让这轮开始?
- **目的**: 解决了什么真问题?
- **状态**: 落地了什么? 缺什么?
- **方法**: 用对了哪几个机制? 错在哪?
- **前提**: 哪些假设现在还对?
- **已知未知**: 哪些缺口暴露了? (诚实暴露, 不藏短)
- **历史**: 跟前几轮比变化在哪?
- **基调**: 这轮整体什么味道? (一句话)

**反模式**: 流水账 / 报喜不报忧 / 缺"已知未知"

## 跳阶段的代价

最容易跳的 3 个阶段:

| 跳过阶段 | 后果 | 实测代价 |
|---|---|---|
| 2 范围 | 直接进 4 设计 | 5 工具一起做, 中途记不清边界 → 边数错 |
| 3 规划 | 直接进 5 实施 | 顺序错位, 阻塞下游的事被压后 |
| 9 回顾 | 写完 commit 完事 | 沉淀不进仓, 下次又踩同坑 |

**经验值**: 每跳一个阶段, 完成度质量降 30%.

## 完整 vs 半跑

| 跑法 | 同样 8.8 分含义 | 适用 |
|---|---|---|
| 9 阶段全跑 | 可发布 | 外部交付 / 钱相关 / 不可逆 |
| 7 阶段跑 (跳 2-3) | 内部可用 | 内部工具 / 频繁迭代 |
| 5 阶段跑 (跳 1, 8, 9) | prototype 验证 | 概念验证 / 一次性 |
| 4 阶段跑 (v0.6 桥接, 跳 1, 2, 3, 8, 9) | 仓内自测 | 框架研究本身 |

## 实证: sas-graph v0.6 → v0.6.1 (回顾)

| 阶段 | 当时做了? | 缺失 | 质量影响 |
|---|---|---|---|
| 1 愿景 | 部分 ("决策 = Edge") | 没说"为什么是 Edge 不是 Node" | -10% |
| 2 范围 | **没做** | 凭感觉选 5 工具, 没 Must/Not/Maybe | -20% |
| 3 规划 | **没做** | 5 工具一起做, 没排序 | -20% |
| 4 设计 | ✅ 4 面展开 | — | 0 |
| 5 实施 | ⚠️ 缺 PIVOT echo | 中途忘了 1 次 (边数错 2 次) | -15% |
| 6 验证 | ✅ vitest 19/19 | — | 0 |
| 7 部署 | ✅ git push | 没写 rollback | -5% |
| 8 监控 | N/A (mini-app) | 未来加 | 0 |
| 9 回顾 | ⚠️ commit msg 一笔 | 没正式 retrospective | -15% |

**扣分合计**: -85% × 8.8 / 9 = -0.83
**实际自评**: 8.8 - 0.83 = **7.97 → 收 8.0**

**结论**: 半跑的状态, 跟 8.8 分"看着像"但实际只有 8.0. 阶段 2+3+9 三个**最该补的顿悟门**各扣 15-20%, 是大头.

## 9 阶段跟 4 步桥接的关系

```
IPL 9 阶段:
1 愿景
2 范围        ←  这 4 个是 4 步桥接没显式拆的
3 规划
4 设计        ← 4 步桥接的"4 面展开"
5 实施        ← 4 步桥接的"制品" (扩)
6 验证        ← 4 步桥接的"多维评分"
7 部署        ← 4 步桥接的"制品发布" (扩)
8 监控        ← 4 步桥接没覆盖
9 回顾        ← 4 步桥接的"PIVOT echo" (扩)
```

**4 步桥接 = IPL 的阶段 4-7 浓缩版**, 适合仓内自测.
**IPL 9 阶段 = 4 步桥接的工程化扩展**, 适合真实项目.
