## User 观察的现象

今晚 25 分钟推了 6 个子版本, 8 分钟推了 v0.6.8 (diffGraphs) — **速度快**。
但每步只做了 1 层工作:
- ✅ **验证** (verification): 跑测试, tsc, demo 跑出效果
- ❌ **推断** (inference): 从已知推未知
- ❌ **判断** (judgment): 评估设计选择

**所以完整度是"平面化的"——只有 1 层深度, 没有 3 层立体**.

## 3 层完整度的具体内容

### 推断 (inference) — 从已知推未知

每步**主动想**:
1. **性能边界**: 大图 (1000 节点) / 大边 (10000 边) 下, 当前实现够快吗?
2. **集成行为**: 跟其他模块 (merge / persistence / render) 行为对齐吗?
3. **跨场景**: 不同输入 (空图 / 完全相同 / 完全不同) 行为正确吗?
4. **未来扩展**: 加 ignore fields / schema 元信息 / smart diff 会破坏什么?

**vs 当前**: 我只想到 "8 个测试覆盖", 没想"10 万节点下能不能跑".

### 判断 (judgment) — 评估设计选择

每步**主动评**:
1. **API 形态**: 当前的 `diffGraphs(g1, g2)` 是最优 API 吗? 应该有 `diffGraphsSymmetric`?
2. **实现选型**: JSON.stringify vs lodash.isEqual vs schema-aware, 哪个更适合 Graph?
3. **错误处理**: 当前没 try/catch, 大图 OOM 怎么办?
4. **可读性**: returned GraphDiff 6 字段太长, summary 是否冗余?

**vs 当前**: 我只实现 "能用", 没问 "为什么这么实现".

### 验证 (verification) — 跑 + 测 + 比对

每步**主动跑**:
1. **runtime 测试**: vitest 通过
2. **compile-time**: tsc --noEmit 通过
3. **integration**: demo 跑出真实效果
4. **边界**: 空 / 单 / 不连通 / 完全相同 / 完全不同

**这是我现在唯一做的 1 层**.

## 反哺 v0.6 / v0.8 框架

### v0.6 (insight-to-work) 加第 6 步 "**判断**"

之前是 4 步 (顿悟→验证→4 面→制品) + 1 步"暴露" (v0.8.1 提).
现在加第 6 步 "**判断**" (v0.8.4 提):

```
6 步 = 顿悟 → 推断 → 验证 → 4 面 → 判断 → 制品
        ↓        ↓       ↓      ↓      ↓       ↓
       WHY    WHAT   HOW    DEEP  WHY-NOT  SHIP
```

每步的"目的":
- 顿悟: WHY (为什么做)
- 推断: WHAT (有什么可能)
- 验证: HOW (怎么测)
- 4 面: DEEP (多角度)
- **判断**: WHY-NOT (为什么选这个, 不选那个)
- 制品: SHIP (交付)

### v0.8 IPL 加阶段 5.5 "**推断**"

之前 9 阶段, 阶段 5 (实施) 没细分.
加 5.5 "推断": 实施前先想"可能暴露什么 / 性能 / 集成".

```
阶段 1 愿景
阶段 2 范围
阶段 3 规划
阶段 4 设计
阶段 5 实施 (含 5.5 推断)
阶段 6 验证
阶段 7 部署
阶段 8 监控
阶段 9 回顾
```

### 6 维自检 (v0.8.2) 加第 7 维 "**判断**"

之前 6 维 (Schema/时间/部署/跨进程/算法/工具链).
加第 7 维 "**判断**" — 评估当前设计选择, 显式列出"WHY-NOT":

```
7 维自检 = Schema + 时间 + 部署 + 跨进程 + 算法 + 工具链 + 判断
```

## 教训

1. **速度快 ≠ 完整度低** — 但**快而不做推断+判断 = 平面化**
2. **每步 3 层 = 立体完整度** — 验证是基础, 推断+判断让每步"立起来"
3. **"为什么" 比 "是什么" 更深** — 判断层问 WHY-NOT, 推断层问 WHAT
4. **框架应该强制 3 层** — v0.6 加第 6 步 "判断", v0.8 加 5.5 "推断"
5. **user 反馈是元观察的源头** — "平面化" 这个观察是我自己看不到的

## 自动化路径

让 LLM 自动跑 3 层, 不靠人补:

### 短期 (v0.8.5)
- commit msg 加 "3 层自检" checkbox:
  ```
  feat: v0.6.9 deep-ignore-fields

  3 层自检:
  - 推断: ✓ (3 个推断列在 PR description)
  - 判断: ✓ (3 个判断: API 形态 / summary 冗余 / entity vs fields)
  - 验证: ✓ (8 测试 + tsc + benchmark)
  ```

### 中期 (v0.9)
- LLM hook: commit 前自动问 3 个 "为什么" (WHY-NOT)
- LLM 自动推 3 个 "可能" (WHAT)
- 跟 v0.8.2 6 维自检 + v0.8.4 3 层完整度 整合

### 长期 (v1.0+)
- LLM 自己跑 3 层, 不靠 commit msg 显式
- 跟 user PIVOT 协同: user 触发 PIVOT, LLM 跑 3 层
