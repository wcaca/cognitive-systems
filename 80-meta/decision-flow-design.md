# Decision Flow Design · 决策流程的设计原则

> [draft → active] 2026-07-01 v0.4
>
> **核心命题**：治理框架不是"事后怎么审"，是"事前/事中怎么决策"。
> 项目阶段决定"决策强度 vs 排查强度"的权重。人类因为精力有限，应该是"校准决策标准"而非"逐一排查决策"。
>
> **本文目标**：把 cognitive-systems 仓之前的所有研究（consistency-governance / cron-as-system-mirror / context-preservation / trend-prediction）整合进一个**统一框架**——决策流程设计原则。

---

## 一、为什么需要这个框架

### 现状（v0.3 之前）

cognitive-systems 仓已经分别研究了：
- v0.2：consistency-governance（事后审计 + 决策反查）
- v0.2.1：cron-as-system-mirror（决策可解释性要求）
- v0.3：context-preservation + trend-prediction（决策的语境化 + 预判）

**但没研究的是**：决策流程**本身**该怎么设计。

这是治理框架的"元层前提"——它假设"决策会发生，要反查"。它没问的是"决策**怎么发生**"。

### 用户的核心命题

> 基于问题（事后排查），相比修问题，更重要的是完善**决策流程**（事前/事中设计）。
> 
> 如果是新开的项目，强化决策相比找问题更重要。
> 如果是成熟项目，决策流程已经稳定，则反之。
> 
> 决策流程中最重要的是 agent 决策流程。
> 人类因为其精力有限性，起着**校准**作用，而非**逐一排查**的作用。

### 这意味着

- **新开期**：决策强度 > 排查强度（决策错了一切白费）
- **成熟期**：决策强度 < 排查强度（流程稳，问题排查能 follow 模式）
- **人类介入**：不是每次决策都审，而是**校准决策标准 + 抽样监督 + 异常响应**

这是治理框架的**统一视角**——把之前的研究都纳入同一框架。

## 二、4 阶段动态权重

| 阶段 | 决策强度 | 排查强度 | 人类介入密度 | 主要介入点 |
|---|---|---|---|---|
| **新开（探索）** | 极高 | 低 | 高（每次关键决策） | 决策标准 / 优先级 / 边界 |
| **早期（建模）** | 高 | 中 | 中高（流程审批） | 决策模式抽象 / 决策粒度调整 |
| **成熟（运营）** | 低 | 中高 | 低（异常处理） | 异常响应 / 重大变更审批 |
| **退化（修复）** | 中 | 极高 | 高（紧急决策） | 优先级 / 风险承受 / 决策暂停 |

### 核心规律

**决策强度 + 排查强度 = 常数（项目可用精力）**

- 流程稳的阶段（成熟）：问题排查能 follow 模式，所以高效 → 排查强度可以高
- 流程弱的阶段（新开）：问题排查是猜测，所以低效 → 决策强度必须高

**harness 不能给所有阶段同一个治理强度**——必须**根据项目阶段动态调整**。

### 项目阶段的判断信号

不是看时间，是看 4 个信号：

| 信号 | 新开 | 早期 | 成熟 | 退化 |
|---|---|---|---|---|
| **决策数量 / 周** | 极少 | 适中 | 极高 | 适中 |
| **决策反查失败率** | 高（无规则） | 中 | 低 | 高（异常） |
| **异常发生率** | 高 | 中 | 低 | 高 |
| **决策模式复用率** | 低（每次新决策）| 中（开始模式化） | 高 | 低（异常打破模式）|

**"决策强度" = 决策重要度的加权总和**，不是简单数次数。

## 三、agent 决策流程的 4 个不变量

无论项目阶段如何变化，agent 决策流程必须有 4 个**不变量**——这是"无论哪一阶段都要设计好的"：

### 不变量 1：决策触发显式化

**问题**：什么时候做决策？
**反模式**：让 agent 自己判断"该不该做"
**正解**：决策触发规则必须显式定义，存在 system memory 里

落地：
```
决策触发 = (条件 A ∧ 条件 B) ∨ (条件 C ∧ 优先级 > 0.7)
         其中 A/B/C 必须在 system memory 里能找到定义
```

**为什么是关键**：决策触发如果不显式，agent 会在不该决策时决策（浪费精力），或该决策时不决策（错过机会）。

### 不变量 2：决策标准显式化

**问题**：按什么标准做决策？
**反模式**：让 agent 用自己的"好恶"判断
**正解**：决策标准写在协议里，**由人类校准**，不是 agent 自动生成

落地：
```
决策标准 = { 安全 > 可恢复 > 性能 > 成本 > 美观 }
         其中顺序和权重由人类校准
```

**关键点**：标准必须**稳定**（不让 agent 自己改），**可校准**（让人类能调权重）。

### 不变量 3：决策信号可观测

**问题**：人类能看见 agent 在做什么决策？
**反模式**：靠人类主动查询（精力不允许）
**正解**：决策信号自动分级上报

落地：
```
决策上报分级：
- L0（日常）：不上报，写 log
- L1（重要）：写 weekly report
- L2（关键）：写 daily report
- L3（不可逆）：立即通知 + 等人类确认
```

**关键点**：人类不应该去"查"，应该让信号"推"过来。

### 不变量 4：决策复盘机制化

**问题**：每个决策都能追溯 + 经验能积累？
**反模式**：决策完了就忘，下次重复犯同样错
**正解**：决策记录机制化，复盘节奏化

落地：
```
decision_log = [
  {
    id, decision, options_considered,
    chosen_standard, result, retrospective
  }
]
→ 月度 review → 季度 audit
```

**关键点**：复盘不是为了"惩罚"，是为了"更新标准"。

## 四、3 个动态权重

这 3 个**随项目阶段调整**：

### 权重 1：决策质量 vs 决策数量

| 阶段 | 倾斜 |
|---|---|
| 新开 | 重决策**质量**（少数决策必须对）|
| 早期 | 重决策**质量** + 决策模式抽象 |
| 成熟 | 重决策**数量**（高频小决策必须高效） |
| 退化 | 重决策**质量**（关键决策必须对） |

**杠杆**：调整"决策粒度"——
- 新开项目 = 粗粒度决策（每次都是大事，少而精）
- 成熟项目 = 细粒度决策（每次都是小事，多而快）

### 权重 2：事前控制 vs 事后检查

| 阶段 | 倾斜 |
|---|---|
| 新开 | 事前控制为主（白名单 + 强制 review） |
| 早期 | 事前 + 事后并行 |
| 成熟 | 事后检查为主（日志 + 异常监控） |
| 退化 | 事后检查为主（异常响应） |

**杠杆**：调整"决策审批流"——
- 新项目 = L1↑ 决策需要人类审批
- 成熟项目 = L3 决策才需要人类审批

### 权重 3：人类介入频率 vs 介入价值

| 阶段 | 频率 | 价值密度 |
|---|---|---|
| 新开 | 高频低密度 | 校准（建标准）|
| 早期 | 中频中密度 | 优化标准 |
| 成熟 | 低频高密度 | 只看异常 |
| 退化 | 低频极高密度 | 紧急指挥 |

**杠杆**：调整"人类介入阈值"——
- 新开 = 任何决策不通过人类都停下
- 成熟 = 异常决策不上报就停下

## 五、人类校准的 3 个具体杠杆

人类精力有限，需要用**杠杆**让介入"以一敌百"：

### 杠杆 1：决策标准校准（最高杠杆）

**什么时候**：项目初期 / 重大变更前
**做什么**：审核 agent 用的决策标准本身
**杠杆倍数**：极大
- agent 用错标准 = 1 万次决策都错
- 校准标准 = 1 次干预影响所有后续

**为什么是最高**：决策标准是"决策的决策"——校准了它，下面的所有决策自动按新标准走。

### 杠杆 2：决策信号抽样（中等杠杆）

**什么时候**：项目稳定期
**做什么**：每周看 N 个随机决策，看是否按标准行事
**杠杆倍数**：中
- 抽样 5% 的决策 = 用 5% 的精力监督 100% 的决策

**为什么中等**：抽样统计有效，但不能纠正偏差——只能发现偏差。

### 杠杆 3：异常响应（低杠杆高紧迫）

**什么时候**：决策失败时
**做什么**：firefighting + 复盘 + 更新决策标准
**杠杆倍数**：低（一次失败只能影响后续 1 个决策）
**紧迫性**：高

**校准的核心 = 把介入预算花在杠杆最高的地方**。

```
新开期 → 主要在杠杆 1（建标准）
成熟期 → 主要在杠杆 2（抽样监督）
退化期 → 杠杆 3（紧急 + 复盘）
```

**这个视图 = "决策流程设计"是 cognitive-systems 仓的真正主线**。

之前的研究都是这条主线的**具体章节**。

## 七、mavis harness 自身应用

### 当前阶段判断

mavis 处于"早期→成熟"的过渡阶段：

- ✅ 已有基础身份机制（session_id / agent_name / parent_session_id）
- ✅ 已有 cron 机制
- ✅ 已有 memory 机制（mavis-memory / agent topics）
- ⚠️ 但决策流程没显式化（什么时候做决策、按什么标准）
- ⚠️ 决策信号不可观测（不知道 mavis 在做什么决策）
- ⚠️ 决策复盘没机制（agent_runs 表里只有结果，没有推理链）

### 4 个不变量在 mavis 的当前覆盖

| 不变量 | 当前 | 缺什么 |
|---|---|---|
| 1. 决策触发显式化 | ⚠️ 部分 | "什么时候决策"写在 prompt 里但没标准化 |
| 2. 决策标准显式化 | ❌ | prompt 里有隐含标准但没抽出来 |
| 3. 决策信号可观测 | ❌ | 没有 decision_log |
| 4. 决策复盘机制化 | ❌ | 没有月度 review |

### 应用建议

**v0.4 行动**：
1. 把"决策标准"显式提取（从 prompt 里挖出来变成可校准的标准）
2. 加"决策信号"通道（每次 L1↑ 决策都生成"决策解释"）
3. 加"决策复盘"机制（每 100 次决策做一次 review）

**v0.5 行动**：
4. 在 mavis 的 cron 任务上加"已知未知 + 决策标准"两个字段（不变量 1+2 的实例化）

## 八、落地检查清单

一个新的 agent harness，要满足以下才算有"基本的决策流程"：

- [ ] 决策触发规则在 system memory 里能找到
- [ ] 决策标准由人类显式定义，不靠 agent 自动学
- [ ] 决策信号分级上报机制（L0-L3）
- [ ] 决策日志格式统一（id, decision, options, standard, result, retrospective）
- [ ] 月度复盘机制
- [ ] 季度 audit 机制
- [ ] 人类介入预算花在"杠杆最高的"地方

## 九、开放问题

1. **怎么自动判断项目当前处于哪个阶段？**
   - 现状：靠人判断
   - 目标：基于历史数据自动判断
   - 难点：阶段过渡是连续的，不是离散的

2. **决策标准该有多"显式"？**
   - 太显式 = 失去灵活
   - 太隐式 = 失去可校准
   - 怎么平衡？

3. **决策信号太多怎么办？**
   - L0 / L1 / L2 / L3 分级后，可能 L2 还是太多
   - 怎么进一步分级 / 聚合

4. **决策复盘的"经验"怎么结构化？**
   - 复盘结果写哪里？
   - 怎么被下次决策自动使用？
   - 这是 context-preservation 和 consistency-governance 的连接点

5. **跨 session 的决策一致性怎么保证？**
   - session A 做了个决策
   - session B 不一定知道
   - 怎么让决策标准跨 session 一致

## 十、一句话总结

> **决策流程设计 = 项目阶段 × 决策强度 × 人类精力 × 杠杆选择**。
> 
> 没有"通用最佳"的决策流程——只有"在该阶段、该阶段任务、该阶段精力下"的**最优**决策流程。
> 
> harness 的责任不是"消除决策"，是"让决策在对的时机、按对的标准、产生可观测的信号、并可被复盘"。
> 
> **这才是治理的真正含义**。
