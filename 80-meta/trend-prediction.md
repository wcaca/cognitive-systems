# Trend Prediction · 对话趋势预判机制

> [draft] 2026-07-01 v0.3
>
> 配套 [`80-meta/context-preservation.md`](./context-preservation.md) 使用。
>
> **核心命题**：LLM 有一种传统研究没有的能力——**从历史对话里学出模式，预测下一个问题**。这种预判能力可以反向影响现在的文件结构，让研究仓**主动服务于未来的对话**。

---

## 一、什么是"对话趋势预判"

### 定义
基于历史对话模式和当前状态，预测接下来的对话可能走向哪些方向。

### 跟传统研究的不同

| 传统研究 | LLM 时代研究 |
|---|---|
| 研究方向靠"立项"决定 | 研究方向受对话引导 |
| 未来读这份研究的人 = 公众 | 未来读这份研究的人 = 公众 + 后续 LLM/agent |
| 文件是"产物" | 文件是"对话的可寻址缓存" |
| 不可预测未来怎么读 | 可以预判接下来会被怎么问 |

**LLM 时代的特殊能力：让"未来读者"（包括 LLM 自己在未来回看）能更高效地找到需要的信息。**

## 二、预判的 3 种类型

### 类型 1：内容预判
**预判下一步会聊什么内容 → 提前在当前文件留好接口**

#### 例子（cognitive-systems 仓的实际应用）

我们聊完"行为一致性治理"后，预判下一步大概率会聊"治理的具体场景"。所以在 `consistency-governance.md` 末尾加了"演化路线"段，列出 v0.3 / v0.4 / v0.5 可能的方向（**不展开**，只是接口预留）。

#### 落地格式
在文件末尾加：
```markdown
## 后续可能的研究方向（占位）
- 治理失败的案例研究
- 治理机制的代价分析
- 治理 vs 创新 的平衡
```

### 类型 2：关联预判
**预判下一步会跨方向 → 提前在仓里建好跨方向的索引**

#### 例子
我们聊完"cron 是全栈镜子"后，预判接下来会聊"mavis 自己怎么改"。所以在 evolution.md 里 v0.3 计划段加了"给 mavis 自己的 cron 任务加'已知未知'清单"。

#### 落地格式
在 00-essence/research-map.md 里加：
```markdown
## 跨方向连接（占位）
- agent-harness ↔ cron → cron-prompt-template（计划 v0.5）
- agent-harness ↔ audit → first weekly audit（计划 v0.6）
```

### 类型 3：反方预判
**预判用户会反问 → 提前准备"反方观点"**

#### 例子
提出 cron 设计的 6 模式时，预判用户会问"那 cron 是不是过设计了"。所以在 `cron-as-system-mirror.md` 里加了"反模式"段，提前说"什么时候 cron 反而是负担"。

#### 落地格式
在文件末尾或独立段加：
```markdown
### 反方观点 (Counter-View)
可能反对：cron 设计是不是过设计了？
回应：当 X 场景时，cron 不如直接对话；当 Y 场景时，cron 是必须的。
```

## 三、预判的反模式（避免过度预判）

### 反模式 1：把预判当承诺
写了"未来研究方向"就**觉得必须做**。结果被自己的预判绑架。
- **修正**：预判是占位，不是承诺。展开是后续 commit 的事。

### 反模式 2：把预判当主文档
在主文档里写满了"未来研究"，反而挤占了当前内容。
- **修正**：预判放末尾或独立段，**不跟主内容混**。

### 反模式 3：过度展开预判
预判的方向写得太详细，结果未来真正做的时候被"前作"绑住。
- **修正**：预判只写 1-2 句话的"方向"，不展开。

### 反模式 4：忽视反方
只预判"用户会顺着想"，不预判"用户会反对"。
- **修正**：每个版本都加"反方观点"段，列出最可能的反对 + 回应。

### 反模式 5：预判后不验证
做了预判，但从来不验证"我预判对了吗"。
- **修正**：每个大版本结束时，回看上一版本的"未来伏笔"，看哪些预判对了、哪些错了。

## 四、预判的工具化

### 工具 1：文件末尾的"未来伏笔"段
**每个 v0.X 的核心文件末尾**留一段：
```markdown
## 后续研究方向（占位，不展开）
- [方向 1]：可能从 [触发点] 演化
- [方向 2]：可能从 [触发点] 演化
```

### 工具 2：跨方向索引（research-map.md）
在 `00-essence/research-map.md` 里维护一个"跨方向连接"段，列出当前已经预判到的跨方向。

### 工具 3：演进元规则
每次大版本结束时（v0.X.0）：
1. 回看上一版本的"未来伏笔"
2. 验证预判对了几个、错了几个
3. 把"被验证的预判"提升为正式研究方向
4. 把"被否证的预判"移到 80-meta/revisions/

### 工具 4：对话密度指标
对话密度 = 单位时间内对话的"深度跳跃次数"。
- 高密度对话后，预判的置信度也应该高
- 低密度对话（闲聊/普通任务），不预判

## 五、跟已有体系的连接

### 跟 context-preservation 的关系
预判 = "主动构造未来语境"。保存 = "记录当下语境"。
- 保存是**被动**（发生了什么记什么）
- 预判是**主动**（我觉得会发生什么，提前准备）
- 两者互补：保存覆盖"已经发生的"，预判覆盖"即将发生的"

### 跟 consistency-governance 的关系
预判也是一种"决策"——决定"未来研究方向"。
- 应该走"决策反查"机制：未来回看时，问"这个预判的依据是什么"
- 应该走"白名单权限"：预判不能修改 system memory（宪法）

### 跟 cron-as-system-mirror 的关系
cron 任务的 prompt 可以利用"对话趋势预判"——
- 预判"今天用户最可能关注什么" → cron 调整产出方向
- 预判"明天可能出错什么" → cron 加预防逻辑
- 这是 cron-as-system-mirror 第 4 层依赖（harness 元认知）的进阶

## 七、开放问题

1. **怎么评估预判的精度？**
   - 现状：没有评估机制
   - 目标：预判对了几个 / 错了几个 的量化指标
   - 难点：预判的"对/错"边界模糊

2. **预判失败的成本怎么算？**
   - 现状：不知道预判失败的代价
   - 目标：知道"过度预判"和"预判不足"哪个更糟
   - 难点：不同语境下成本不同

3. **怎么让 LLM 在生成内容时主动预判？**
   - 现状：靠人写"未来伏笔"
   - 目标：LLM 自动在每个文件末尾生成预判段
   - 难点：怎么避免 LLM 写"假预判"（看似有道理实际没意义）

4. **预判 vs 创造的边界在哪？**
   - 预判是"准备未来"
   - 创造是"决定未来"
   - 怎么避免预判变成"代用户决定"？

## 九、一句话总结

> **预判不是算命，是"为对话的下一拍做准备"**。
> 
> LLM 时代的特殊能力：基于历史对话模式预判下一步，主动构造"未来语境"。
> 把预判沉淀成机制 = 让研究仓**主动服务于未来对话**，而不是被动记录过去。
> 
> **预判 + 保存 = 完整的研究语境**。
