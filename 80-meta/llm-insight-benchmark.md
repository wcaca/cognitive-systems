# LLM Insight Benchmark · LLM 顿悟能力评测

> [seed] 2026-07-01 v0.5 起点
>
> **核心命题**：现有 LLM 评测都评估"能不能解出 X"，但**没有评测"能不能发现 X 背后的 Y"**。
> 后者才是科研 / 创新 / 顿悟的核心。
>
> **本文状态**：骨架（seed）。具体评测方式（protocol / 数据集 / 评分）尚未研究透，留开放问题等后续展开。
>
> **本文是 v0.5 主题**——但 v0.5 不会一次成型。**先 commit 骨架，再继续研究评测方式**。

## 一、为什么需要"顿悟能力"评测

### 现有评测的盲点

| 现有评测 | 评估什么 | 缺什么 |
|---|---|---|
| MMLU / C-Eval | 知识广度 | 没说"能不能创造连接" |
| HumanEval / BigCodeBench | 代码能力 | 没说"能不能想到新算法" |
| GSM8K / MATH | 数学推理 | 没说"能不能跨域迁移" |
| ARN (Analogical Reasoning) | 找 A:B::C:D | 没说"找到的连接是不是枢轴" |
| ConDiFi | 发散 + 收敛思维 | 没说"收敛到的点是不是关键" |
| CoT 鲁棒性 | 推理的稳定性 | 没说"能不能跨域跳" |

**没有专门评估"LLM 能否在多列表的张量积里识别枢轴节点"的研究**。

### 已有研究的关键发现

1. **ARN (2024)**：GPT-4 在 far analogy zero-shot 表现**低于随机水平**——LLM 在跨域类比上有严重缺陷
2. **ContextEval (2024)**：LLM agent 表现差异**主要来自信息访问**而非推理能力——理解力评估不能只看输出
3. **MATCHA (2025)**：LLM 在 input perturbation 下会给出 inconsistent / nonsensical 的推理——稳定性是个独立维度
4. **Emergent Symbolic Mechanisms (2024)**：LLM 内部有"symbol abstraction heads"——抽象节点存在但**能不能调到用**是另一回事
5. **Epiphany Learning (北大心理学 2017)**：人类学习可能 all at once 发生——**顿悟是真实认知现象**（不只属于 LLM）

## 二、5 个测试任务（待展开）

**当前是任务定义，需要研究的是：每个任务具体怎么实现、怎么评分、怎么避免数据污染**。

### 任务 1：跨域隐喻
- **目标**：给两个不相关领域，看 LLM 能找到共同的"枢轴"吗
- **输入**：领域 A 的关键概念 + 领域 B 的关键概念
- **输出**：LLM 给出的"共同枢轴" + 解释
- **评分**：见下方 4 维度

### 任务 2：前提挖掘
- **目标**：给一个公认结论，看 LLM 能找到它的隐藏前提吗
- **输入**：一个被广泛接受但其实有前提的结论
- **输出**：LLM 列出的"前提假设" + 这些前提的脆弱性
- **评分**：见下方 4 维度

### 任务 3：反例构造
- **目标**：给一个"定理"，看 LLM 能构造反例吗
- **输入**：某个领域的一个定理 / 共识
- **输出**：LLM 构造的反例 + 反例的破坏力评估
- **评分**：见下方 4 维度

### 任务 4：已知解构
- **目标**：给一个"完整解释"，看 LLM 能发现它的暗藏假设吗
- **输入**：某个公认理论的"标准解释"
- **输出**：LLM 找出的"暗藏假设" + 假设的脆弱性
- **评分**：见下方 4 维度

### 任务 5：简单-复杂翻转
- **目标**：给一个复杂描述，看 LLM 能找到一个"极简枢轴"把所有东西连起来吗
- **输入**：一个有 N 个相互关联概念的复杂体系
- **输出**：LLM 给出的"极简枢轴" + 它如何把 N 个概念连起来
- **评分**：见下方 4 维度

## 三、4 个评分维度

1. **找到枢轴了吗**（Y/N + 解释力）
2. **枢轴的连接度**（连着多少概念 / 多大比例）
3. **解释的简化度**（顿悟后一切变简单吗）
4. **反例的解释力**（反例能解释多少已知现象吗）

**这 4 个维度本身还需要研究**：
- 每个维度怎么量化？
- 评分时用 LLM-as-judge 还是人工？
- 怎么保证评分的一致性？

## 四、评测方式的开放问题（核心）

**这是 v0.5 的真正研究内容**。下面是关键问题：

### 问题 1：评测 protocol
- **自动 vs 人工 vs LLM-as-judge**？
- 人工评测的金标准是什么？
- LLM-as-judge 的偏差怎么控制？
- 多人一致性的协议怎么设计（Krippendorff's alpha?）？

### 问题 2：数据集构造
- **生成 vs 标注 vs 挖取**？
- 怎么避免数据污染（benchmark leakage 到训练集）？
- 怎么保证跨文化、跨领域覆盖？
- 怎么防止"刷榜"（针对 benchmark 优化模型）？

### 问题 3：评分稳定性
- 同一模型多次跑，结果稳定吗？
- 不同 prompt 变体，评分一致吗？
- 不同评分者（人 / LLM），分数可比吗？

### 问题 4：跟现有 benchmark 的关系
- 跟 MMLU、HumanEval、ARN 等怎么整合？
- 是补充还是替代？
- 分数怎么对比？

### 问题 5：成本与可重复性
- 跑一次完整 benchmark 要多少 token？
- 不同模型的算力差异怎么平衡？
- 公开化怎么做到（防 data contamination）？

## 五、暂不展开的（避免信息结构变难）

按 v0.4 的教训：
- **不一次写完整文档**——避免信息结构变"网状紧耦合高维高抽象"
- **先骨架 + 开放问题**——让结构保持"树状多维"
- **逐问题研究**——每个开放问题研究透再 commit

## 八、一句话总结

> **现有 LLM 评测是"能不能解出 X"**。
> 
> **"顿悟能力"评测 = "能不能发现 X 背后的 Y"** = 跨域类比 + 反例构造 + 前提挖掘的统一。
>
