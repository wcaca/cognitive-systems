## 审计的 3 个目的

1. **发现**：agent 是否在按当前协议行事？
2. **追溯**：上周的某个决策是基于什么做的？现在还能复现吗？
3. **改进**：哪些机制在实践中没用上？哪些机制缺失？

**没有审计的治理 = 自欺欺人。**

---

## 审计的 4 个层次

### Level 0：自动化 log 检查（每日）
- 所有 agent session 必须有统一 log 格式
- 每条 log 包含：session_id, agent_name, action, timestamp, decision_hash
- 自动化工具每天扫一遍：
  - 是否有 agent 触发 L3 动作无人类确认
  - 是否有 agent 修改 system memory
  - 是否有协议指纹不一致
- **异常 → 自动告警**

### Level 1：决策抽样（每周）
- 抽样上周 5-10 个关键决策
- 检查每条决策：
  - 是否有签名链（agent_id / context_hash / signed_at）？
  - 决策依据是否引用了具体协议版本？
  - 是否有备选方案记录？
- 抽样报告写进 `80-meta/audits/<week>.md`

### Level 2：协议一致性（每月）
- 对比本月所有 agent 加载的协议指纹
- 不一致 → 立即定位 + 修复
- 检查协议 CHANGELOG 是否有"无文档"的改动

### Level 3：深度审计（每季）
- 挑一个研究方向 / 一个子系统做深度 review
- 检查治理机制本身的演进：
  - 硬约束是否被绕过？
  - 软机制是否在实践中被遵守？
  - 反向防御是否被演练过？
- 输出"治理机制的健康度报告"

## 审计的反模式

### 反模式 1：审计流于形式
只检查"有没有 log"，不看"log 内容是否合理"。
→ 审计必须有"语义检查"，不只"格式检查"。

### 反模式 2：审计由 agent 自己执行
agent 审 agent = 自审自。
→ 审计必须有外部性：人类、peer agent、或独立工具。

### 反模式 3：审计报告无人读
写了 100 份审计报告，没人看。
→ 审计必须进入"决策流"：异常 → 修复 → 关闭。

### 反模式 4：审计成为权力工具
审计用来抓"违规"而不是改"机制"。
→ 审计目的是**改系统**，不是罚 agent。

---

## 审计跟"决策反查"的关系

- **决策反查**：单次决策的反查（修改旧判断时写明理由）
- **行为审计**：一段时间内所有决策的回顾

两者互补：
- 反查是"事中"记录
- 审计是"事后"检查

---

## 审计的元规则

> **审计机制本身要被审计。**

谁来审计审计者？
- Level 0 自动化的"自动化审计" → 由人类季度检查
- Level 1-2 周月审计 → 由 peer agent 交叉审计
- Level 3 季度深度审计 → 由人类或独立 agent

**这条规则防止"审计者变成新的独裁者"。**
