# Consistency Governance · 行为一致性治理

> [draft] 2026-07-01 v0.2
>
> **核心问题**：如何在 agent 行为一致性的层面上维护整个系统，防止不同时期、不同工作、不同能力下的 agent 扰乱、引导错误这个系统？
>
> **本文立场**：这不是技术问题，是治理问题。技术能做"硬约束"，治理要做"软机制"，最后一道防线是"反向防御"。三层叠起来，才有可能让一个长跑型系统在 agent 群里不被腐蚀。

---

## 一、"一致性"的 4 个层面

传统软件里这 4 个都被类型系统 / 数据库 / 版本控制解决了。Agent 系统里这 4 个**全部需要重新设计**——因为 agent 是非确定的、上下文有限的、会"自作主张"的。

| 层面 | 问题 | 不解决会怎样 |
|---|---|---|
| **身份一致** | 今天的 agent 是昨天的 agent 吗？它怎么证明？ | 每个 agent 都按自己理解"我是谁"，系统分裂 |
| **协议一致** | 不同 agent 看到的"系统宪法"是同一份吗？ | agent A 按 v1 协议行事，agent B 按 v2，行为冲突 |
| **决策一致** | 类似情况，不同 agent 会做类似决策吗？ | 同类问题反复出现，每次答案不同 |
| **记忆一致** | A 写下的记忆，B 读出来还是原意吗？ | 记忆被解读成不同含义，行为漂移 |

## 二、硬约束（架构层）— 违反触发告警

### 约束 1：协议文档只读化
- 系统核心协议对 agent **只读**
- Agent 想改协议 → 写 PR，人类审批
- 协议有显式版本号（v1.2.3），agent 必须知道自己加载的是哪个版本
- 每次协议改动留 git commit，**改动内容必须可被 diff**

### 约束 2：核心判断的签名链
每个关键决策（修改协议、清空记忆、删除资产、对外发消息）必须有**审计记录**：

```
{
  decision_id,
  agent_id,
  agent_version,
  context_hash,      // 基于上下文的指纹，证明"我是基于什么想的"
  decision,
  signed_at
}
```

谁、什么时候、基于什么上下文、做了什么决定 — 缺一个就是欠债。

### 约束 3：记忆三层隔离

```
user memory      ← 用户的偏好/事实，agent 可读，写要授权
agent memory     ← agent 的工作笔记，agent 可写但有版本
system memory    ← 系统的"宪法"，agent 只读，变更走流程
```

**禁止任何 agent 直接修改 system memory。** 这是"宪法不被篡改"的硬保证。

### 约束 4：核心机制不变量
定义一组**不可变的核心**：
- 身份怎么生成
- 记忆怎么存
- 协议怎么加载
- 决策怎么记录

这些核心机制**不参与版本演进**，只能扩展不能修改。任何想"重构身份机制"的 agent → 直接拒绝。

## 三、软机制（流程层）— 日常治理

### 机制 1：变更窗口与观察期
- 破坏性变更必须发生在**特定窗口**（低峰期）
- 变更后必须留 **24-72h 观察期**，期间系统处于"双协议并行"状态
- 观察期内任何异常 → 自动回滚

### 机制 2：决策反查（Decision Backtracking）
Agent 修改旧判断时，必须主动写明：
- 旧判断是什么
- 什么新证据 / 新推理推翻了它
- 这告诉我们研究方法本身有什么问题

**不写反查理由的"静默修改"= 危险信号。** 这条机制同时保护了"知识演进"和"防篡改"。

落地格式：
```
80-meta/revisions/<revision-id>.md
├── old_judgment
├── new_evidence
├── new_judgment
└── methodological_implication
```

### 机制 3：协议演进 snapshot
每次协议改动留 snapshot：

```
30-protocols/
├── v1.2.3/
│   └── snapshot.md
├── v1.2.4/
│   └── snapshot.md
└── CHANGELOG.md   ← "v1.2.3 → v1.2.4 改了什么、为什么"
```

新 agent 加载协议时，必须能查到"这个协议从哪来、经历过什么"。

### 机制 4：多 agent 一致性协议
对**关键决策**（不是所有决策）：
- 至少 2 个 agent 独立判断
- 达成一致才执行
- 不一致 → 上报人类

这避免了"单个 agent 抽风把系统带歪"。代价是慢，但只在关键决策上启用。

### 机制 5：行为审计
定期抽样：
- 上周 agent 做了哪些决策
- 这些决策是否符合当前协议
- 有没有"自作主张"的痕迹

审计结果写进 `80-meta/audits/`。**审计本身要有签名**，证明它真的被做过了。

详见 [`audit-protocols.md`](./audit-protocols.md)

## 四、反向防御（兜底层）— 最坏情况怎么救

### 防御 1：硬 reset 按钮
保留一个**人类独占的 emergency reset**：
- 把所有 agent memory 备份
- 把协议回滚到上一个 stable 版本
- 重启所有 agent session
- 通知所有相关方

**这个按钮不能被 agent 触发**，只能人类触发。这是"系统不会失控"的最后保险。

### 防御 2：协议指纹
每次 agent 启动时记录"我加载的协议指纹"（hash）。如果发现不同 agent 加载的协议指纹不一致 → 立即告警。

```
{
  agent_id,
  protocol_version,
  protocol_hash,
  loaded_at
}
```

### 防御 3：白名单优于黑名单
- **白名单**：定义 agent 允许做的动作集合（"这 100 件事可以"）
- **黑名单**：定义禁止的动作集合（"这 10 件事绝对不行"）

**白名单优于黑名单**——黑名单总有遗漏，白名单"不在名单内就不许做"。

权限层级（参考 creation-loop v0.3 的实践）：
- **L0 read** — 只读，不修改任何东西
- **L1 append** — 只追加，不删不改
- **L2 modify** — 可修改自己产生的
- **L3 one-way door** — 不可逆动作（推送、删除、对外），必须人类确认

### 防御 4：决策可解释性
每个 agent 决策必须能回答：
- "你为什么这么做？" → 引用了哪个协议 / 哪条记忆 / 哪个判断
- "还有别的方案吗？" → 必须列出至少 1 个备选
- "你确定吗？" → 必须给出置信度

**不可解释的决策 = 不被执行的决策。**

## 五、人机协作的边界

agent 不应该独自决定的事：
- 修改系统协议
- 删除任何资产
- 清空任何记忆
- 对外发不可撤回消息
- 公开任何内容

这些事情必须有**人类检查点**。Agent 可以提建议、写草案、写 PR，但执行必须人类点。

## 六、实证：creation-loop v0.3 的治理实践

> 来源：wcaca/creation-loop v0.3 部署记录（2026-07-01）

creation-loop v0.3 在实践中实现了上述机制的大部分：

| 治理机制 | creation-loop v0.3 实现 | 状态 |
|---|---|---|
| 权限分层（白名单）| `autonomy.md` 4 级（L0 read / L1 append / L2 modify / L3 one-way door）| ✅ 已实现 |
| 节奏治理（变更窗口）| `scheduler.md` 4 节拍（daily / weekly / monthly / quarterly）| ✅ 已实现 |
| 全局闸（紧急 reset）| `state.md` mode 开关（auto / paused / manual）| ✅ 已实现 |
| 跨 session 一致性 | cron + state mode 联动（1 个 cron: loop-daily-creator）| ⚠️ 部分实现 |
| 多 agent 一致性协议 | n-of-n 决策 | ❌ 未实现 |
| 决策反查文档化 | revisions 机制 | ❌ 未实现 |
| 协议指纹 | 未实施 | ❌ 未实施 |

### 实战教训

creation-loop v0.3 部署时遇到的**真实问题**：
> mavis cron session 与 root session 不同 sandbox，写不进 /workspace/creation-loop/。

这就是"多 session 一致性"的现实考验。4 个候选 workaround：
1. git-based（写完后 commit/push）
2. communicate-back（通过 mavis communicate 把变更传回 root session）
3. hybrid
4. host cron（绕过 mavis，直接系统级 cron）

**v0.3 的验收标准 = 至少 1 个 workaround 实测能写。**

这个案例完美说明了：治理机制不是设计出来就完事的，**部署时的环境约束会反过来逼迫你重新设计**。

## 七、落地检查清单

一个新的 agent harness，要满足以下才算有"基本的一致性治理"：

- [ ] 协议只读 + 版本号 + git 跟踪
- [ ] 关键决策签名链（至少包含 agent_id、context_hash、signed_at）
- [ ] 记忆三层隔离（user / agent / system）
- [ ] 核心机制不变量清单（哪些不能改）
- [ ] 决策反查文档化流程
- [ ] 多 agent 关键决策一致协议（n-of-n）
- [ ] 行为审计频率和报告位置
- [ ] 人类独占的 emergency reset 流程
- [ ] 协议指纹机制
- [ ] 决策可解释性要求
- [ ] 白名单权限模型
- [ ] L0-L3 权限分层
- [ ] 人机决策边界清单

每缺一条 → 一致性欠债 +1。

## 九、一句话总结

> **agent 系统的行为一致性是治理问题，不是技术问题。**
> 硬约束（架构） + 软机制（流程） + 反向防御（兜底）= 三层缺一不可。
> 没有这治理框架的长跑型 agent 系统，**一定会在 6 个月内被自己的 agent 群腐蚀**。
