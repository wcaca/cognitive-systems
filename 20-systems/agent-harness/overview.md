# Agent Harness 体系总览

> [draft → active] 2026-07-01 v0.1

## 核心问题

**怎么构建一个完整、严密、适应性强、可恢复性强的 agent harness？**

## 三个角色

```
   建筑师 agent                    居民 agent
   (Architect)                     (Resident)
        │                              │
        │   设计、维护                  │   跑在 harness 里
        │   harness 本身               │   执行任务
        ▼                              ▼
   ┌─────────────────────────────────────┐
   │           Harness                    │
   │  (身份/记忆/工具/调度/通信/观测/     │
   │   安全/成本/版本/恢复)               │
   └─────────────────────────────────────┘
                    ▲
                    │ 监测 / 诊断 / 修复 / 升级 / 回滚
                    │
              维护维度 (10 个)
```

### 1. 建筑师 agent
设计 harness 的人（或 agent）。需要：失败想象力、协议思维、分层、不可逆敏感、运维代入、极简偏好。

详见 [`architect.md`](./architect.md)

### 2. 居民 agent
跑在 harness 里的 agent。需要：自观察、诚实、契约纪律、边界行动、记忆连续性、资源自觉。

详见 [`resident.md`](./resident.md)

### 3. 维护维度
harness 作为产品要维护的 10 个能力维度。

详见 [`maintenance-dimensions.md`](./maintenance-dimensions.md)

## 角色-维度矩阵

|        | 身份 | 记忆 | 工具 | 调度 | 通信 | 观测 | 安全 | 成本 | 版本 | 恢复 |
|--------|------|------|------|------|------|------|------|------|------|------|
| 建筑师 | 设计协议 | 抽象 | 注册机制 | 调度模型 | 通道设计 | metric 设计 | 权限模型 | 配额策略 | 兼容策略 | 快照策略 |
| 居民 | 会话身份 | 记忆读写 | 工具调用 | 任务声明 | peer/上级 | 自报状态 | 守权限 | 自查 | 接受升级 | checkpoint |
| 维护 | session 管理 | 状态治理 | I/O 治理 | 调度治理 | 消息治理 | 监控告警 | 审计 | 预算 | 迁移 | 灾备 |

## 一句话原则

> **把"被救"当一等公民，把"刚上线好用"当副产品。**
