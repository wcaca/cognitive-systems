# Cron as System Mirror · Cron 是全栈镜子

> [draft] 2026-07-01 v0.2.1
>
> **核心观察**：cron 不是一个功能，是一个"全栈镜子"。一个 cron 任务能不能跑好，不是 cron 的问题，是**整个系统栈的快照**。
>
> **本文立场**：把 cron 看作 agent harness 治理框架的"集成测试"——cron 失败 = 系统全栈失败。设计 cron 的过程就是暴露系统短板的过程。

---

## 一、为什么 cron 是镜子

| 其他 agent 任务 | cron 任务 |
|---|---|
| 可以"渐进对话"补全上下文 | **一次 prompt，一次响应** |
| 可以追问用户 | **没有追问机会** |
| 可以中途切换策略 | **prompt 写错就全错** |
| 失败可被人发现 | 失败可能**沉默几天**才被发现 |
| 决策可被回顾 | 决策发生在**凌晨 3 点** |

**普通对话的失败 = 单点失败。Cron 任务的失败 = 全栈失败。**

这意味着：cron 跑不好时，根因几乎不在 cron 本身，而在基础设施、模型能力、prompt、harness 元认知、人机耦合中的某一层（或多层）。

---

## 二、cron 暴露的 5 层依赖

```
┌─────────────────────────────────────────────┐
│ cron 任务结果（看起来只是个定时器）           │
├─────────────────────────────────────────────┤
│ 提示词设计     ← 必须自完备（没有追问机会）  │
├─────────────────────────────────────────────┤
│ harness 自我认知 ← "我知道我什么不知道"     │
├─────────────────────────────────────────────┤
│ 人的认知行为耦合 ← 节奏/格式/决策点对齐人脑 │
├─────────────────────────────────────────────┤
│ LLM 能力       ← 理解力、推理、遵循指令     │
├─────────────────────────────────────────────┤
│ 已有体系       ← runner/sandbox/重试/状态  │
└─────────────────────────────────────────────┘
```

### 第 1 层：已有体系（基础设施）

cron 没有基础设施 = 写了个 crontab 但没人接。必须先有：
- runner（谁来跑）
- sandbox（在哪跑）
- error handling（挂了怎么办）
- retry / dead-letter（重试 / 死信）
- state persistence（状态怎么存）
- observability（怎么观察它）

**经验案例**：wcaca/creation-loop v0.3 部署时遇到 cron session 与 root session 不同 sandbox 写不进工作区。这是第 1 层没铺好的典型表现。

### 第 2 层：LLM 能力

cron 任务没有"渐进对话"——它就一次 prompt 一次响应。模型必须能：
- 在长 prompt 里不丢关键指令
- 在没有"上下文补全"的情况下做出正确判断
- 遇到没见过的输入不胡来

**模型能力天花板 = cron 任务的能力天花板**。

### 第 3 层：提示词设计（比对话 prompt 难 10 倍）

cron prompt 是**没机会追问的自完备文本**。意味着：
- 必须显式列所有边界
- 必须显式列所有错误处理
- 必须显式列"做不了就上报"的路径
- 不能依赖"agent 灵机一动"

判断 cron prompt 的标准：
- [ ] 没有"你觉得"——必须"你必须"
- [ ] 没有"通常"——必须"总是"
- [ ] 边界条件全部列出
- [ ] 失败处理全部列出
- [ ] 输出的格式有显式 schema

### 第 4 层：harness 的已知未知（关键中的关键）

cron 任务最容易失败的方式是：
> "harness 不知道这个任务不知道自己不知道什么"。

具体表现：
- 任务以为某个工具可用，结果环境里没装
- 任务以为某个数据存在，结果是空
- 任务以为上次的"修复"还在，结果被别的 agent 改回去了
- 任务以为自己的 prompt 是最新版本，结果 harness reload 的是老版本

**harness 必须对自己的"已知未知"（known unknowns）有元认知**。这一层是 traditional 编程里不存在的——程序不会"以为"自己知道什么，程序只知道 type signature 里写的东西。

每个 cron 任务应该有一个**已知未知清单**：
```
known_unknowns:
  - "我不知道今天网络是否能访问外部 API"
  - "我不知道用户今天关注什么"
  - "我不知道前 7 天的决策是不是还有效"
```

任务开始时**主动探测**这些未知，发现问题就上报而不是装作知道。

### 第 5 层：人的认知行为耦合

cron 服务的对象是人。如果 cron 的工作节奏 / 产出格式 / 决策点跟人脑不匹配，再好的 cron 也是噪音：
- 每天 9 点推 50 条 summary → 没人看
- 半夜 3 点做决策 → 人不在 → 决策悬空
- 推的内容跟人当下关注点不对齐 → 价值归零

**这一层决定了"做对的事"和"做不该做的事"的边界。**

具体的设计要求：
- **节奏对齐**：cron 频率 vs 人脑能消化的频率
- **时间窗口**：什么时候推 vs 人什么时候有空看
- **决策权限**：cron 任务能自动做 vs 必须等人类
- **噪音控制**：重复内容抑制、相关度排序

## 三、5 层之间的耦合关系

不是简单的"底层断 → 顶层断"。5 层之间是相互影响的：

**自下而上的硬约束**：
- 第 1 层断 → 任何上层都跑不起来
- 第 2 层能力差 → 上层 prompt 写得再好也白搭

**自上而下的设计约束**：
- 第 5 层不匹配 → 下面 4 层做得再好也是浪费
- 第 4 层不清晰 → cron 任务不知道自己什么时候"够好"

**横向的循环依赖**：
- 第 3 层 prompt 引用了第 4 层的"已知未知清单"——两者必须同步演进
- 第 2 层 LLM 能力升级 → 第 3 层 prompt 可以写得更简，但第 4 层元认知要求可能更高

**一个失败模式：第 1 层看起来 ok，但实际是脆的**。比如 sandbox 偶尔能写偶尔不能写——这比"完全不能写"更糟，因为它会在最糟糕的时候失败。

## 四、cron 设计的反模式

### 反模式 1：把 cron 当 crontab
> "我写个 crontab 就行"。

错。crontab 只能告诉你"在 X 时间跑了 Y 命令"，不能告诉你"跑得对不对、结果有没有用"。**crontab 是 5 层依赖里最浅的一层**。

### 反模式 2：cron prompt 跟对话 prompt 同款
> "我跟我对话的 prompt 写得挺好的，复用一下就行"。

错。对话 prompt 假设有"追问机会"，cron prompt 没有。直接复用 = 任务会沉默失败。

### 反模式 3：沉默失败
> "cron 没输出就是没问题"。

错。**沉默失败是最危险的失败**。必须有：
- 成功有输出
- 失败有报警
- "无输入/无输出"也是一种失败（"我不知道"）

### 反模式 4：让 cron 做决策
> "反正 3am 没人，cron 自己决定"。

错。**人不在 ≠ 决策可以被绕过**。人不在时，cron 应该**等**或**上报**，不是**擅自做**。

### 反模式 5：无限重试
> "失败就重试，反正便宜"。

错。重试掩盖了真问题。**重试次数本身就是设计参数**——超过 3 次还在重试 = 设计问题，不是网络问题。

### 反模式 6：cron 任务写自己
> "让 cron 修自己的 bug"。

错。**任务不能修改自己**——这是 consistency-governance 里"协议只读"原则的 cron 场景化。

## 五、cron 设计的模式

### 模式 1：自完备 prompt 模板
每个 cron prompt 必须包含：
```
1. 角色定义（我是谁）
2. 任务定义（要做什么）
3. 输入定义（基于什么数据）
4. 工具定义（能用什么、不能用用什么）
5. 边界定义（什么情况停下不上报）
6. 输出定义（schema 是什么）
7. 错误处理（出错了怎么办）
8. 已知未知清单（我可能不知道什么）
9. 决策权限（我能不能自动做 X 决策）
10. 自我验证（我怎么验证我做得对）
```

### 模式 2：双轨决策
- **轨道 A：cron 任务自己跑**，产出"待人类 review 的草稿"
- **轨道 B：人类确认或修改**后才是正式决策

**cron 永远不直接做不可逆动作**。它只能产出"建议"，等人类确认。

### 模式 3：节奏对齐
- **轻量级 cron**（5-15min）：探测、汇总、健康检查
- **中量级 cron**（小时-天）：报告、归档、清理
- **重量级 cron**（周-月）：总结、决策建议、长期演化

**人脑能消化的最大频率 = 每天 3-5 条结构化推送**。超过 = 噪音。

### 模式 4：观察-上报双信号
- **观察信号**：cron 任务日常产出的状态、metric
- **上报信号**：cron 任务遇到"我搞不定"时的紧急通知

观察信号走 dashboard（人想看时看），上报信号走 IM（推送到人面前）。

### 模式 5：cron 任务的版本控制
- cron prompt 必须有版本号
- 每次 prompt 改动留 git commit
- 改动要说明"为什么改、影响什么任务"

**没有版本控制的 cron prompt = 漂移的根源。**

### 模式 6：cron 任务的"已知未知"清单
任务启动时主动探测：
- 我依赖的工具是不是还在？
- 我依赖的数据是不是还是新鲜的？
- 我依赖的协议版本是不是还是当前版本？
- 我之前的"成功标准"还是不是当前的成功标准？

任何一个变了 → 主动暂停 + 上报。

## 六、跟 consistency-governance 的连接

| 治理机制 | 在 cron 场景的具体考验 |
|---|---|
| **决策反查** | cron 任务改了旧判断，谁发现？怎么回滚？ |
| **白名单权限** | cron 默认能用什么工具、什么权限（L0-L3）？ |
| **决策可解释性** | cron 任务做了一件事，harness 能复现它的推理吗？ |
| **多 agent 一致性** | cron 任务的决策需要 n-of-n 验证吗？ |
| **行为审计** | cron 任务的行为被审计了吗？ |
| **协议只读** | cron 任务能修改 system memory 吗？（不能） |
| **三层记忆隔离** | cron 任务能写 user memory 吗？（要授权） |
| **核心不变量** | cron 任务能改"调度机制"本身吗？（不能） |

**cron 是治理框架的"集成测试"**——治理机制设计得行不行，跑一个生产 cron 任务就知道了。

## 七、研究问题清单（开放）

以下问题还没有满意答案，记下来作为后续研究方向：

1. **怎么自动生成 cron 任务的自完备 prompt？**
   - 现状：每个 cron 任务都要人手写
   - 目标：基于任务描述 + 历史数据 + 失败模式，自动生成"几乎完备"的 prompt
   - 难点：怎么验证"几乎完备"？

2. **怎么检测 cron 任务的"沉默失败"？**
   - 现状：很多 cron 任务挂了没人知道
   - 目标：harness 自动检测"任务没产出 ≠ 任务成功"
   - 难点：怎么区分"任务没事干"和"任务挂了"？

3. **怎么把 cron 任务的失败变成系统改进的输入？**
   - 现状：失败被 retry 掩盖
   - 目标：每次失败自动产出"这次告诉我们系统哪里有短板"
   - 难点：怎么避免被同一类失败反复打扰？

4. **怎么让 cron 任务"知道"自己的 prompt 不是最新版本？**
   - 现状：prompt 改了但跑的还是老的
   - 目标：cron 任务启动时主动校验 prompt 版本
   - 难点：harness reload 机制需要支持版本感知

5. **怎么评估 cron 任务对人的"价值密度"？**
   - 现状：cron 任务推的东西多但有用的少
   - 目标：每次产出都标注"这条对人的价值是什么"
   - 难点：价值是主观的，怎么量化？

6. **cron 任务之间怎么协作？怎么避免互相干扰？**
   - 现状：多个 cron 任务可能同时改一个文件
   - 目标：cron 任务有锁、有依赖图、有冲突检测
   - 难点：跨 session 锁的实现

7. **怎么让 cron 任务"模仿人"的工作节奏？**
   - 现状：cron 按固定时间跑
   - 目标：cron 根据"人今天的关注点"调整自己
   - 难点：怎么让 harness 知道"人今天的关注点"

## 八、给 mavis harness 自己的建议

我们自己在用的 mavis harness 也有 cron 任务（如 loop-daily-creator）。对照本框架看，**有 3 个明显的欠债**：

1. **没有自完备 prompt 模板**：cron prompt 跟对话 prompt 没严格区分
2. **沉默失败没有兜底**：cron 任务挂了只能等下次人发现
3. **没有"已知未知"清单**：cron 任务启动时不主动探测依赖

**v0.3 改进建议**：
- 写一份 `cron-prompt-template.md` 作为标准模板
- 在 mavis harness 里加 cron 健康检查（每 5 分钟扫一遍，看哪个任务最近 24h 没产出）
- 在 cron 注册时强制要求填"known unknowns" 字段

---

## 九、一句话总结

> **cron 是 agent harness 的"集成测试"——它跑得好 = 系统全栈 ok；它跑不好 = 系统一定有哪层断了。**
> 
> 设计 cron 的过程不是写 prompt 的过程，是**给整个系统做体检**的过程。
> 
> 一个好的 cron 任务不是"能跑"，是"在 5 层依赖都脆弱的情况下依然能跑、对、且不沉默失败"。
