# 60-tools

> **[active v0.8.19]** — 工具契约层 (scripts/ 里反复使用的工具抽到这里写契约)

> **顿悟 V 起源 (v0.8.19)**：T 协议 (填实顺序协调) 实施时发现, scripts/ 里的工具没人写 README, 想给别人用都得读源码。把"反复使用的工具"契约化 = 工具发现的最后一公里。

> **路径不动, 契约抽出来** — 工具本体仍在 scripts/ (避免破坏现有调用), 60-tools/ 只放契约 + 例子 + 故障排查。

## 已收录

- `cross-repo-status.md` — 跨仓元数据同步脚本契约 (v0.8.19 / 顿悟 V 第一篇)

## 收录标准

满足以下任一条件, 就该有契约:

1. **跨仓使用** — 不只一个仓用, 不写契约别人接不住
2. **cron 触发** — 自动化跑, 失败需要排查路径
3. **复杂 flag** — 5+ flag 或参数解析复杂
4. **历史踩坑多** — 至少有 1 个已知坑 (从 git log 翻 commit msg)

## 不收录 (避免噪音)

- **一次性脚本** — 用完即弃, 没必要契约化
- **trivial 包装** — 5 行内的 git command 包装
- **私有实验** — 不属于公共知识库的

## 何时更新契约

- 工具加新 flag → 必更新契约的"调用契约"表
- 工具 bug 修复 → 必更新"已知坑"段 (避免下次踩同样坑)
- 工具弃用 → 必把契约移到 `90-conventions/deprecated-tools.md`, 60-tools/ 移除

## 与其他目录的关系

- `scripts/` — 工具本体 (bash / python / js)
- `60-tools/` — 工具契约 (本文)
- `30-protocols/` — 协议 (工具可能依赖的协议)
- `70-artifacts/` — 工具产出物 (例如 cross-repo-status-archive/)

## 顿悟 V 的边界

只抽**已经被反复使用的工具**的契约, 不主动给所有 scripts/ 里的脚本写 README。
- 用法频率 = git log 里出现次数 + issue 提及次数
- 没被反复用 = 没人看, 写契约是浪费
