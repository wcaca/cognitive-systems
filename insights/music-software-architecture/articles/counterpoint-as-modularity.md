# 对位法即模块化 · 300 年前的软件架构

> v0.11 完整 prototype artifact
> 起源: cognitive-systems benchmark v0.11 第 3 案例
> 方法: insight-to-work 4 步流程 (v0.6 框架)

## 缘起

在研究 cognitive-systems v0.6 (insight-to-work 框架) 时, 我被 Bach 的赋格曲吸引.
Bach 一个 4 声部赋格, 每个声部独立, 严格协和音程规则, 但整体听起来像"一个完整作品".
**这不就是软件模块化吗?**

不是. **这就是软件模块化**.

## 核心命题

> 音乐对位法 (Bach 时代) 和软件模块化 (现代 CS) 本质是同一种系统.
> 三个机制同时存在才能"无中心协调下的全局和谐":
> 1. **独立**: 每个声部 / 模块有独立状态
> 2. **规则约束**: 对位规则 / API 契约
> 3. **局部错误隔离**: 错音不影响其他声部 / bug 不影响其他模块

## 5 个跨域映射

| 音乐 | 软件 | 共同点 |
|---|---|---|
| 声部 | 模块 | 独立单元 |
| 协和音程 | API 契约 | 全局规则 |
| 通过音 | 中间件 | 暂时冲突但解决 |
| 终止式 | 事务提交 | 全局一致点 |
| 转调 | 协议升级 | 规则动态变化 |

## 3 个具体案例

### 1. Unix 哲学 ≈ Bach 风格
- 小工具 (声部) + 管道 (协和音程) + 文本接口 (调式)
- "做一件事做好" = "1 个声部做 1 个旋律"

### 2. 微服务 ≈ 爵士即兴
- 服务 (乐器) + API 网关 (节奏) + 故障熔断 (错音解决)
- 强解耦 + 弱协调

### 3. 区块链 ≈ 文艺复兴复调
- 节点 (声部) + 共识协议 (对位规则) + 分叉 (终止式变体)
- 完全无中心 + 强数学约束

## 反思: 无中心协同

对位法教会我们 1 个事:
**"协同"不是"控制"的反义, 而是"控制"的一种特殊形式** —— 强规则约束下, 每个单元"自由地"按规则运行, 整体涌现出"协调"。

软件工程的 30 年, 从单体到 SOA 到微服务到 serverless, 都是在"对位法化"。

## 对未来的预测

接下来 10 年最热的系统: **"对位法化" 的多 agent AI 系统**。
- 多个 LLM agent 各自推理 = 多声部
- 明确的输出契约 (API) = 协和音程
- 局部错误隔离 (try-catch + 重试) = 通过音
- 全局规则约束 (宪法 / system prompt) = 调式

这不是比喻, 是设计模式.

## 一句话

> Bach 在 1722 年写《平均律钢琴曲集》时, 已经在做"软件架构"了.
> 只是介质是音符, 不是代码.
