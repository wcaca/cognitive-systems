## 历史面 (History Face)

### 起源
- **1953**：Watson & Crick 发现 DNA 双螺旋结构
- **1958**：Crick 提出分子生物学的"中心法则"

### 演化（4 个阶段）
1. **1960s-70s**：遗传密码破译 → 知道 DNA 是信息载体
2. **1980s-90s**：PCR 技术 → 体外复制 DNA
3. **2000s**：基因测序成本下降 → 大数据时代
4. **2011+**：CRDT 形式化 → 软件"半保留同步"开始有理论基础

### 类比
- **抗体机制**（生物）vs **错误检测码 ECC**（计算机）—— 都是"出错时校验"
- **酶修复**（生物）vs **自动重试 + 事务回滚**（软件）—— 都是"出错时自修复"

✅ **历史面达标**：起源 + 4 阶段演化 + 类比

## 结构面 (Structure Face)

### 上位（更大框架）
- **分布式系统理论**（Lamport 1982 时间逻辑 / CAP 定理）
- **信息论**（Shannon 1948 冗余编码）
- **可靠性工程**（冗余 → 鲁棒性的基本原理）

### 下位（可拆解）
1. **DNA：每条链是另一条的镜像** + 互补配对（A-T, G-C）
2. **软件：双数据库主从** / 多副本（每个副本是另一份的镜像）
3. **git：每个 commit 是另一个 branch 的衍生 + 可合并**
4. **CRDT：每个副本能独立更新 + 最终收敛**
5. **Paxos/Raft：多副本投票 → 一致性**

### 平行（同构邻居，4 个）
- **免疫系统抗体多样性**：B 细胞 + 抗体 = 多副本 + 多样性 = 鲁棒性
- **大脑神经网络冗余**：每个记忆多副本存储 → 抗遗忘
- **生物种系多样性**：每个物种有多个个体 → 抗灭绝
- **multi-version concurrency control (MVCC)**：数据库事务多版本 → 抗冲突

✅ **结构面达标**：上位 + 下位（5 个具体机制） + 平行（4 个同构邻居）

## 应用面 (Application Face)

### 场景 1：协同编辑（CRDT）
- **问题**：多人同时编辑文档如何同步
- **应用**：Yjs / Automerge 等 CRDT 库
- **成功案例**：Figma、Mattermost、Notion 都用了 CRDT

### 场景 2：分布式数据库
- **问题**：多个数据中心写同一份数据
- **应用**：Cassandra / CockroachDB / Spanner
- **成功案例**：Google Spanner 跨数据中心强一致

### 场景 3：版本控制（git）
- **问题**：多人协作开发同一份代码
- **应用**：git 的分支 + 合并 + 冲突解决
- **成功案例**：全球开源生态

### 反例（什么时候不能用这个 connection）
- **强实时控制**（如自动驾驶）：副本同步延迟 = 灾难，反例用集中控制
- **法律/合规敏感场景**：副本创建 = 多版本管理 = 法律风险
- **强一致性场景**（如银行转账）：不能容忍最终一致

### 操作步骤（具体怎么用）
1. 识别"什么可以多副本"（数据 / 配置 / 日志）
2. 选择副本同步策略（同步 / 异步 / 最终一致）
3. 实现冲突检测（CRDT / vector clock / 版本号）
4. 实现冲突解决（LWW / 优先级 / 人工）
5. 设计恢复机制（单副本失败时的重建）
6. 监控副本一致性（lag metric）
7. 定期 chaos engineering 测试副本独立性

✅ **应用面达标**：3 场景 + 1 反例 + 7 步操作

## 限制面 (Limitation Face)

### 边界条件 1：副本间网络可达
- **满足**：副本能定期同步
- **不满足**：脑裂（split-brain）风险

### 边界条件 2：冲突解决策略全局一致
- **满足**：所有副本用同一策略
- **不满足**：副本可能漂移到不同状态

### 边界条件 3：状态空间有界
- **满足**：CRDT 可收敛
- **不满足**：状态爆炸 / 内存爆掉

### 失效场景
- **脑裂（split-brain）**：副本各自为政，无法合并
- **Cascading replication**：一个失败引发所有失败
- **State divergence**：长时间不同步的副本无法收敛

### 误用风险（3 个具体场景）
1. **用"多副本"作为"高可用"替代品**：多副本不解决所有故障类型
2. **用"半保留同步"掩盖架构缺陷**：底层架构问题仍存
3. **误以为"副本 = 0 风险"**：副本增加系统复杂度，本身是新风险源（Cap theorem）

✅ **限制面达标**：3 边界条件 + 1 失效场景 + 3 风险
