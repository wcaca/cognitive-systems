## 引言

生态学和软件工程看似无关，但深究起来，两个领域都在解决同一个问题：**当系统被打扰时，怎么在没有中央控制的情况下恢复平衡？** 这种"自我修复 / 自稳态机制"是连接两者的枢轴节点。

## 三个核心论点

### 1. 自我修复不是"装了监控 + 自动恢复"那么简单

传统软件架构把"自动化"等同于"自动恢复脚本"。但真正的自我修复有三个**必要条件**：

- **冗余**：必须有备份能力（生态中的物种多样性，软件中的多副本）
- **负反馈**：必须有"失败 → 信号 → 调整"的回路（生态中的捕食者-猎物关系，软件中的 circuit breaker）
- **可观测**：失败必须**被系统本身**检测到，不是外部监控

### 2. 自适应系统的关键是"无中心协调" vs "集中控制"

传统软件思路：集中调度 → 中央大脑 → 协调
生态 + 现代自愈系统思路：去中心化 → 局部规则 → 涌现

**这两种思路的根本分歧**：人要不要做"中央协调者"。

| 集中控制 | 自组织 |
|---|---|
| 假设系统是简单 | 假设系统是复杂适应 |
| 中央可预测 | 局部可预测，全局涌现 |
| 失败 = 中心失效 | 失败 = 单点失效但能恢复 |

### 3. 借鉴生态原理设计 resilient 软件

具体可借鉴的 7 个机制：

1. **多样性 → 鲁棒性**（生态） → **多技术栈**（微服务）
2. **分解者**（生态） → **自动回收**（k8s）
3. **捕食者-猎物反馈**（生态） → **circuit breaker**
4. **生态系统迁移**（生态） → **服务自动迁移**
5. **物种共生**（生态） → **sidecar 模式**
6. **自然演替**（生态） → **canary deployment**
7. **稳态机制**（生态） → **SLO + 自动告警**

## 4 个陷阱（必须避开）

### 陷阱 1：把"自修复"当成"不需要运维"
- 仍然需要人观察自修复是否如预期工作
- 自修复失败时需要人介入

### 陷阱 2：自修复逻辑本身出错
- 真实案例：Facebook 2021 BGP 路由故障 — 自修复逻辑本身成为故障源
- 解法：自修复机制本身也要有**外部监控**

### 陷阱 3：掩盖根本架构问题
- 自修复可能是"症状缓解"，但根本架构问题仍在
- 解法：定期 review 自修复频率，过高的频率 = 架构问题

### 陷阱 4：跨过自修复的边界
- 金融 / 医疗 / 安全 场景不能用完全自修复
- 必须有"自修复可达"的边界设计

## 总结

"自我修复 / 自稳态机制"是这个 connection 的核心。**它不是工具，是哲学**——它假设系统具有内在的适应能力，而不是被外部完全控制。

跨域借鉴生态原理，不是为了复制生物学，而是为了**理解一种不同的工程哲学**。
