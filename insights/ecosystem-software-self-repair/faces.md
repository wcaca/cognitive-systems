## 历史面 (History Face)

### 起源
- **1972**：James Lovelock 提出盖亚假说（地球作为自我调节系统）
- **1970s**：复杂系统理论开始成熟（Santa Fe Institute）

### 演化
- **2001**：IBM 提出 autonomic computing，把"自管理"概念引入 IT
- **2010s**：microservices 普及，chaos engineering 出现（Netflix Chaos Monkey）
- **2020s**：自适应数据库（如 CockroachDB 自愈）

### 类比
- 中医"调和"思想 vs 现代控制论 homeostasis
- 道家"无为而治" vs 集中控制 vs 自组织

✅ **历史面达标**：起源 + 演化（4 阶段）+ 类比

## 应用面 (Application Face)

### 场景 1：微服务架构
- **问题**：单个服务故障不要引发雪崩
- **应用**：circuit breaker + chaos engineering
- **成功案例**：Netflix 通过 Chaos Monkey 主动制造故障，提高鲁棒性

### 场景 2：自适应数据库
- **问题**：节点故障时仍能服务
- **应用**：自动分片 + 副本重建 + 共识
- **成功案例**：CockroachDB、TiDB、Cassandra

### 场景 3：CDN 自动故障转移
- **问题**：某节点宕机时用户无感知
- **应用**：健康检查 + DNS 切换 + 边缘缓存
- **成功案例**：Cloudflare、Akamai

### 反例（什么时候不能用）
- **金融交易系统**：必须审计 trail，**不能**完全自修复
- **医疗系统**：必须有医生主导，**不能**自动决策
- **关键基础设施控制**：必须有冗余 + 人工备份

### 操作步骤（具体怎么用）
1. 识别"自修复可达"的边界（什么可以自修复）
2. 设计反馈回路（必须可观测）
3. 加 circuit breaker 隔离失败
4. 加 retry + backoff 恢复
5. 加监控 + 告警（人知道有自修复发生）
6. 定期 chaos engineering 测试

✅ **应用面达标**：3 场景 + 1 反例 + 6 步操作

## 限制面 (Limitation Face)

### 边界条件 1：扰动在范围内
- **条件**：局部失败 + 失败率 < 阈值
- **超过**：自修复机制可能失效（雪崩）

### 边界条件 2：必须有反馈信号
- **条件**：可观测的失败信号
- **不满足**：系统不知道要修复

### 边界条件 3：恢复时间 < 用户等待时间
- **条件**：恢复必须快
- **不满足**：用户感受不到"自修复"，只感受到"宕机"

### 失效场景
- **雪崩式 cascading failure**：所有节点同时失败
- **生态崩溃**：物种大规模灭绝
- **软件崩盘**：cascade bug 让所有机制失效

### 误用风险
- **风险 1**：把"自修复"当成"不需要运维"——实际仍需监控
- **风险 2**：过度依赖自修复掩盖了根本问题（如架构缺陷）
- **风险 3**：自修复逻辑本身出错 → 系统级错误（如 Facebook 2021 BGP 故障）

✅ **限制面达标**：3 边界条件 + 1 失效场景 + 3 风险
