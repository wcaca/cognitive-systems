# Insight Extraction Protocol · 顿悟提取协议

> [draft] 2026-07-04 · v0.8.16 · S 顿悟的实施产物
>
> 配套：[`80-meta/skeleton-as-completeness-signal.md`](../80-meta/skeleton-as-completeness-signal.md) §五
>
> **问题**：当你和 LLM 对话完，发现"顿悟了"，怎么把这个顿悟**可靠地**从对话记录提取出来，变成可以 commit 的 markdown？

---

## 1. 为什么需要协议

LLM 对话里的"顿悟"有 3 个特点：

1. **散落**：跨多轮对话，不在某一轮
2. **隐式**：用户和 LLM 都知道"这个点很重要"，但没说"这是顿悟 X"
3. **可伪造**：LLM 容易把"普通的提了一句"包装成"顿悟"——顿悟密度高 ≠ 顿悟质量高

不协议化的提取 = 凭印象 = 同一段对话，下次提取得不同。

---

## 2. 提取协议：5 步

### Step 1: 标记顿悟时刻
在对话记录中插入 marker：
```
[INSIGHT-START: YYYYMMDD-N]
[INSIGHT-END]
```
N 是当日第几个顿悟（1, 2, 3...）

### Step 2: 写 trigger
顿悟发生前 1-3 轮的具体内容（用户的哪句话 + LLM 的哪句回应）
**必须是原话引用**，不是改写。

### Step 3: 写核心命题
一句话，最多两句话：
"X 是 Y"（顿悟 = 短句）
**不能写**：长段解释、长段类比、长段延伸（这些是 "扩展"，放在另一节）

### Step 4: 写证据
3 类证据之一或组合：
- **内推证据**：从已知推出新结论（"因为 A，所以 B"）
- **实证证据**：做了 / 看到了 / 跑通了（"v0.8.15 跑通后..."）
- **反例证据**：某个看起来应该反例但不是的（"X 看起来反例，但是 Y"）

**至少 1 类**，最好 2 类。

### Step 5: 命名
- 用一个**单字母 + 序号**（P, Q, R, S, T, ...）作为顿悟 ID
- 在 evolution.md / README 中引用时用 ID + 版本号（"v0.8.14 顿悟 Q"）

---

## 3. 协议的反模式

| 反模式 | 表现 | 修正 |
|---|---|---|
| 没有 marker | 凭印象提"昨天好像有个点" | 必须先有 [INSIGHT-START] |
| trigger 写改写 | "用户问了个问题" | 必须原话 |
| 核心命题写长 | 写 5 句话 | 必须 ≤ 2 句 |
| 证据只有"我觉得" | 没有实证 | 至少要有内推或实证 |
| 命名随意 | "那个顿悟" | 必须 ID + 顺序 |

---

## 4. 协议的实施：commit message 模板

```bash
git commit -m "feat(80-meta): v0.8.X <一句话核心命题> = 顿悟 <ID>

- 协议: 30-protocols/insight-extraction-protocol.md §2 5步
- trigger: 原话引用
- 证据: ...
- 命名: 顿悟 <ID> (<单字母>)"
```

---

## 5. 协议的限制

- 这个协议**不能保证顿悟是真的顿悟**——它只是规范提取过程
- 顿悟质量仍然靠 `80-meta/llm-insight-evaluation-protocol.md` v0.5.1 的 5 级评分
- 但没有这个协议，5 级评分就无从评起（因为顿悟都没被提取出来）
